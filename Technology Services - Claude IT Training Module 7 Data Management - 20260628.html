7.2 Analyzing CSV and JSON Data

Sharing data files with Claude is straightforward. The method depends on which interface you're using.

🖥 Desktop App

Drag and drop CSV or JSON files directly into the chat window. Claude reads the file automatically — no copy-paste required.

Best for: Files up to several MB, interactive analysis sessions

⌨ Terminal CLI

Pipe file contents directly into Claude using standard shell syntax.

PowerShell
cat data.csv | claude "summarize this dataset"

Best for: Automation, large files, scripted workflows

🔗 VS Code Extension

Open the data file in the VS Code editor, then reference it in the Claude sidebar panel. Claude can see the active editor's contents in context.

Best for: Developers working alongside code that processes the data

Example workflow: Analyzing a server asset inventory

  1. Export the asset list to CSV from your asset management tool (e.g., Lansweeper, Snipe-IT, or a custom CMDB). Make sure to anonymize any fields containing personal data before proceeding.

  2. Drop the CSV into Claude Desktop. Drag the file from File Explorer into the Claude chat window. You'll see a file attachment chip appear above the message field confirming the upload.

  3. Ask your analysis question in plain language. You don't need to describe the file structure — Claude reads the headers automatically.

  4. Review Claude's structured summary and follow up with refinements: "Now break that down by building location" or "Export just the EOL systems as a table."

7.3 Writing SQL Queries

Generating correct SQL — especially for joins, subqueries, and aggregations — is one of Claude's strongest data capabilities. You describe what you need in English; Claude writes the query.

Context Tip
Always tell Claude your database platform (SQL Server, PostgreSQL, MySQL, SQLite) at the start of the conversation. Syntax differences matter — GETDATE() is SQL Server, NOW() is MySQL/PostgreSQL, datetime('now') is SQLite.

Example 1: Inactive account audit query

What you say to Claude:

Claude
You
SQL Server. Write a SQL query to find all user accounts that haven't logged in for 180 days, joined with their department from the departments table. Include days inactive and sort by most inactive first.
SQL — SQL Server
SELECT
    u.username,
    u.email,
    u.last_login,
    d.department_name,
    DATEDIFF(DAY, u.last_login, GETDATE()) AS days_inactive
FROM users u
LEFT JOIN departments d ON u.department_id = d.id
WHERE u.last_login < DATEADD(DAY, -180, GETDATE())
    AND u.is_active = 1
ORDER BY days_inactive DESC;

Example 2: Query optimization

If a query is timing out or running slow, paste it into Claude and ask for help. Claude does not just fix the syntax — it explains why the query is slow and what index would help.

Claude
You
This query is timing out on our helpdesk database (PostgreSQL, ~2M rows in tickets). Can you optimize it?

SELECT * FROM tickets WHERE LOWER(subject) LIKE '%network%' AND created_at > '2026-01-01' ORDER BY created_at DESC;
Claude

There are two performance problems here:

  1. LOWER(subject) LIKE '%network%' — The leading wildcard (%network%) prevents any index on subject from being used. PostgreSQL has to scan every row and apply the function. This is the primary bottleneck.
  2. SELECT * — Fetching all columns forces a heap fetch on every matching row, even if you only need a few fields.

Optimized approach:

-- 1. Create a GIN index for full-text search (run once)
CREATE INDEX idx_tickets_subject_fts
    ON tickets USING GIN(to_tsvector('english', subject));

-- 2. Optimized query using full-text search
SELECT id, subject, status, assignee_id, created_at
FROM tickets
WHERE to_tsvector('english', subject) @@ plainto_tsquery('english', 'network')
    AND created_at > '2026-01-01'
ORDER BY created_at DESC;

The GIN index enables sub-millisecond full-text matching. Expected improvement: from 8–15 seconds to under 100ms on 2M rows. The index creation itself takes roughly 30–60 seconds and can be done without locking the table using CREATE INDEX CONCURRENTLY.

7.4 Data Transformation Scripts

When data arrives from one system in the wrong shape for another, Claude writes the transformation script. This is faster than building it manually and produces clean, commented code you can re-run.

Example: Normalizing phone numbers from an AD export

Prompt to Claude: "Write a PowerShell script to read a CSV of Active Directory user exports, normalize all phone numbers to E.164 format (+1XXXXXXXXXX), and output a clean CSV. Handle missing and malformed entries gracefully."

PowerShell — Normalize-PhoneNumbers.ps1
# Normalize-PhoneNumbers.ps1
# Reads AD user export CSV and normalizes phone numbers to E.164 format.
# Usage: .\Normalize-PhoneNumbers.ps1 -InputFile users_export.csv -OutputFile users_normalized.csv

param(
    [Parameter(Mandatory)][string]$InputFile,
    [Parameter(Mandatory)][string]$OutputFile,
    [string]$PhoneColumn = "telephoneNumber"
)

function ConvertTo-E164 {
    param([string]$RawPhone)
    if ([string]::IsNullOrWhiteSpace($RawPhone)) { return "" }

    # Strip all non-digit characters
    $digits = $RawPhone -replace '\D', ''

    switch ($digits.Length) {
        10 { return "+1$digits" }                          # North American 10-digit
        11 { if ($digits[0] -eq '1') { return "+$digits" } }  # Already has country code
        default { return $RawPhone }                       # Return as-is if unrecognized
    }
    return $RawPhone
}

$data   = Import-Csv -Path $InputFile
$errors = 0
$fixed  = 0

$normalized = $data | ForEach-Object {
    $original = $_.$PhoneColumn
    $e164     = ConvertTo-E164 -RawPhone $original
    $_.$PhoneColumn = $e164
    if ($original -ne $e164 -and $e164 -ne "") { $fixed++ }
    if ($e164 -eq "" -and $original -ne "") { $errors++ }
    $_
}

$normalized | Export-Csv -Path $OutputFile -NoTypeInformation
Write-Host "Processed $($data.Count) records."
Write-Host "  Normalized: $fixed"
Write-Host "  Skipped (empty or unrecognized): $errors"
Write-Host "Output written to: $OutputFile"

Python alternative for larger datasets

For datasets above ~50,000 rows, Python with pandas is faster and more memory-efficient than PowerShell CSV processing.

Python — normalize_phones.py
import pandas as pd
import re

def normalize_phone(phone):
    """Convert phone number to E.164 format (+1XXXXXXXXXX)."""
    if pd.isna(phone) or str(phone).strip() == "":
        return ""

    digits = re.sub(r'\D', '', str(phone))

    if len(digits) == 10:
        return f'+1{digits}'
    elif len(digits) == 11 and digits[0] == '1':
        return f'+{digits}'

    return phone  # Return as-is if unrecognized

df = pd.read_csv('users_export.csv')

# Track what changed
original = df['phone'].copy()
df['phone_normalized'] = df['phone'].apply(normalize_phone)

changed = (original != df['phone_normalized']).sum()
empty   = (df['phone_normalized'] == "").sum()

df.to_csv('users_normalized.csv', index=False)

print(f"Processed {len(df):,} records")
print(f"  Normalized:            {changed:,}")
print(f"  Empty/unrecognized:    {empty:,}")
print(f"  Output: users_normalized.csv")

Other common transformation tasks Claude handles well

  • Date format normalization: Converting mixed date formats (MM/DD/YYYY, YYYY-MM-DD, "June 28 2026") to a consistent ISO 8601 format
  • Column renaming and reordering: Mapping fields between two system export formats
  • Merging multiple CSVs: Combining monthly ticket exports into a single file, deduplicating on a key column
  • JSON flattening: Converting nested JSON API responses to a flat CSV for import into Excel or a database
  • Encoding repair: Fixing UTF-8 mojibake in legacy exports from older systems
Iteration Tip
Paste the first few lines of your actual CSV (with sensitive data removed) when asking for a transformation script. Claude will match the exact column names and handle edge cases specific to your data.

7.5 Log Analysis

Server logs, event logs, and application logs are verbose and time-consuming to parse manually. Claude reads log excerpts and extracts what matters: errors, patterns, anomalies, and timelines.

Effective log analysis prompts

Web Server Logs

"Here are the last 500 lines of our Apache access log. Are there any unusual patterns or IP addresses making an unusually high number of requests?"

Claude will group by IP, flag request rates, identify suspicious user agents, and spot 4xx/5xx spikes.

Syslog / System Events

"Find all ERROR and CRITICAL entries in this syslog export and group them by service. Which service has the highest error rate?"

Claude will extract, count, and categorize errors — something that would take grep and awk pipelines manually.

Windows Event Log

"This Windows Event Log export has 2,000 events. Summarize what happened between 2:00 AM and 4:00 AM — I'm investigating a performance incident."

Claude builds a timeline, flags warnings and errors, and correlates events that occurred close together.

Application Traces

"Here is a stack trace from our ticketing system. What caused this exception and what file/line should I look at first?"

Claude reads the full stack trace and points directly to the root cause, not just the surface error.

CLI approach: Pipe logs directly from PowerShell

You don't need to export to a file first. Pipe live output directly into Claude from the terminal:

PowerShell — Live log analysis
# Analyze the 100 most recent system errors
Get-EventLog -LogName System -EntryType Error -Newest 100 |
    ConvertTo-Csv |
    claude "summarize these system errors and identify any recurring issues or patterns"
PowerShell — Application log file
# Send a log file excerpt to Claude for analysis
Get-Content "C:\Logs\IIS\u_ex260628.log" -Tail 500 |
    claude "identify any 500 errors, slow responses over 5 seconds, and the top 5 requesting IPs"
Bash (Linux) — Syslog analysis
# Last 200 lines of syslog, filtered to errors and criticals
grep -E "ERROR|CRITICAL|WARN" /var/log/syslog | tail -200 | \
    claude "group these log entries by service and summarize the top issues"
Size Considerations
Claude's context window can handle several thousand lines of log data comfortably. For very large log files (100,000+ lines), pre-filter with grep or Select-String before sending to Claude. Always remove or mask IP addresses and usernames from externally shared logs.

What Claude produces from log analysis

  • A structured summary of error types and their frequency
  • A timeline of events with timestamps highlighted
  • Identification of which services or components are most affected
  • Actionable next steps: which log to check next, which service to restart
  • A condensed incident report you can paste into a ticket

7.6 Data Documentation and Dictionaries

Data without documentation creates institutional knowledge gaps. When staff leave or systems are handed over, undocumented schemas become liabilities. Claude generates data dictionaries in minutes from schemas or sample data.

Prompts for data documentation

  • "Here is the schema for our helpdesk database. Generate a data dictionary with field names, types, nullable status, and plain-English descriptions."
  • "Write descriptions for each column in this CSV — the file is a student enrollment extract from Banner."
  • "Document this API JSON response — what does each field represent? Include the data type and example values."
  • "Generate a README for this database table explaining what it stores, how it relates to the users table, and when records are created vs updated."

Workflow Tip
Keep the generated data dictionary as a living document. When the schema changes, paste the updated CREATE TABLE statement and ask Claude: "Update the data dictionary to reflect these changes." Claude will diff the changes and revise only the affected rows.

7.7 Working with Excel Data

Excel is ubiquitous in university administration. Claude integrates with Excel workflows in several ways, even without a direct Excel plugin.

Methods for using Claude with Excel

📋 Paste Table Data Directly

Copy cells from Excel and paste into Claude chat. Claude reads tab-separated or space-aligned table text and understands the structure automatically.

Best for: Quick analysis of small to medium tables

📄 Export to CSV First

File → Save As → CSV. Then drag the CSV into Claude Desktop or pipe it via CLI. No data is lost for flat sheets.

Best for: Large datasets, complex analysis

📏 Ask for Excel Formulas

Describe what you want to calculate and Claude writes the formula, including complex nested functions.

Best for: Formula generation without manual lookup

🤖 Generate Automation Scripts

Ask Claude to write Python (openpyxl or pandas) or PowerShell scripts to process Excel files programmatically.

Best for: Recurring reports, batch processing

Excel formula generation examples

What You Ask For What Claude Produces
"VLOOKUP to match employee IDs in column A of Sheet1 against Sheet2 and return the department name" =VLOOKUP(A2,Sheet2!$A:$B,2,FALSE)
"Count tickets in column C that are 'Critical' and were created after June 1" =COUNTIFS(C2:C1000,"Critical",D2:D1000,">"&DATE(2026,6,1))
"Average resolution time in hours where status is 'Resolved'" =AVERAGEIF(F2:F1000,"Resolved",G2:G1000)*24
"Flag rows where the last login date is more than 180 days ago" =IF(TODAY()-B2>180,"Inactive","Active")

Python script for automated Excel processing

Python — Generate monthly summary from helpdesk Excel export
import pandas as pd
from openpyxl import load_workbook
from openpyxl.styles import Font, PatternFill, Alignment
from datetime import datetime

# Load the helpdesk export
df = pd.read_excel('helpdesk_export_june2026.xlsx', sheet_name='Tickets')

# Parse dates
df['created_at'] = pd.to_datetime(df['created_at'])
df['resolved_at'] = pd.to_datetime(df['resolved_at'])
df['resolution_hours'] = (df['resolved_at'] - df['created_at']).dt.total_seconds() / 3600

# Monthly summary by category
summary = df.groupby(['category', 'priority']).agg(
    ticket_count=('id', 'count'),
    avg_resolution_hrs=('resolution_hours', 'mean'),
    sla_breaches=('sla_breached', 'sum')
).round(1).reset_index()

# Write to Excel with formatting
with pd.ExcelWriter('monthly_summary.xlsx', engine='openpyxl') as writer:
    summary.to_excel(writer, sheet_name='Summary', index=False)

    ws = writer.sheets['Summary']
    header_fill = PatternFill("solid", fgColor="004077")  # Acadia Blue
    for cell in ws[1]:
        cell.font = Font(bold=True, color="FFFFFF")
        cell.fill = header_fill
        cell.alignment = Alignment(horizontal='center')

    for col in ws.columns:
        ws.column_dimensions[col[0].column_letter].width = 20

print(f"Summary written: {len(summary)} rows across {summary['category'].nunique()} categories")
print(f"Report saved: monthly_summary.xlsx")

7.8 Data Safety Rules

Critical — Data Privacy Requirements

Claude processes the data you share in your conversation. That data is subject to Acadia University's data handling policies and applicable privacy legislation. Following these rules is mandatory, not optional.

What you must never share with Claude

Data Category Why It's Protected What to Do Instead
Student academic records (grades, enrollment, transcripts) FERPA, Acadia privacy policy Anonymize: replace student IDs with "Student-001", remove names
Personal Health Information (PHI) PIPEDA, PHIA (Nova Scotia) Remove all health-related fields before sharing any export
SINs, financial account numbers, passwords Identity theft risk, PCI DSS Never include. Mask as "***-***-***" before sharing
Live production credentials (API keys, DB passwords) Security risk Replace with placeholder values like YOUR_API_KEY_HERE
Full names + contact info together PII linkage risk Share names OR contact info, not both — or replace with "User A", "User B"

For analysis requiring real sensitive data

  1. Use Claude Enterprise or on-premises deployment. Acadia's Enterprise agreement includes data privacy provisions that cover processing of institutional data. Confirm with Technology Services management which tier your use case requires.

  2. Consult the Privacy Officer. For any use case involving student or employee records, contact Acadia's Privacy Officer before proceeding. They can advise on whether a Privacy Impact Assessment is required.

  3. Review Anthropic's data handling documentation. Understand the data retention policy for your Claude plan — Consumer, Team, and Enterprise plans have different data residency and retention terms.

Safe and Still Useful
Anonymized data is still analytically useful. Claude can identify patterns, generate scripts, and produce documentation from representative fake data or fully anonymized samples — you don't need real personal information for most IT analysis tasks.
