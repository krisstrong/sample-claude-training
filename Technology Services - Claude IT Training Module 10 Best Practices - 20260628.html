10.1 Claude Is a Powerful Tool — Use It Responsibly

Throughout this training series you have seen what Claude can do: write production-quality scripts, review security vulnerabilities, analyze logs, draft communications, and manage documentation at a pace and scale that would be impossible alone. That capability is real, and it comes with real responsibility.

Claude amplifies what you bring to it. A well-formed prompt with accurate context produces genuinely useful output. A prompt that includes bad assumptions, sensitive data, or insufficient context produces output that can be wrong — or worse, output that exposes information that should be protected. With IT-level permissions, Claude can make real changes to real systems. This module covers the discipline that keeps that power safe and effective.

Claude's Output Is Only as Good as Your Input

Claude does not know your network topology, your specific configurations, or your institutional context unless you tell it. The more accurate and complete the information you provide, the more reliable the output.

Garbage in, garbage out still applies. Test Claude's suggestions before you act on them.

Every Output Requires Human Review

Claude is a first draft, not a final authority. Code must be tested. Commands must be verified. Communications must be reviewed. Documentation must be checked for accuracy.

You are the expert. Claude is the assistant. The authorization to act is always yours.

Core Principle
With IT permissions, Claude can make changes to production systems on your behalf. The human in the loop is not a formality — it is the essential safeguard. Always understand what a command or script does before running it, regardless of whether Claude wrote it.

10.2 What NOT to Share with Claude

Important — Before You Share Anything
Never share the following with Claude unless you are operating under a formal enterprise data privacy agreement. Conversation text is transmitted to Anthropic's servers and may be retained under standard consumer and professional plan terms. If in doubt, anonymize or omit the sensitive element and describe it in general terms instead.
Category Examples Why It Matters
Passwords & credentials Admin passwords, API keys, SSH private keys, service account passwords, database connection strings with credentials embedded Claude stores conversation text. Credentials pasted into a conversation can be extracted from conversation history and may be retained by Anthropic under standard plan terms.
Student personal data Names combined with student IDs, grade records, enrollment status, academic standing, financial aid details FERPA protected. Disclosing student educational records to unauthorized parties — including AI systems without a formal data processing agreement — is a legal violation.
Employee personal data Social Insurance Numbers, health information, performance reviews, salary details, disciplinary records PIPEDA (Canada's federal privacy law) restricts how personal information about individuals can be collected, used, and disclosed. AI conversations are a disclosure.
Acadia proprietary configurations Production database connection strings (without credentials removed), internal IP address maps, firewall rule sets, detailed network diagrams Sharing infrastructure details creates security risk. This information could facilitate targeted attacks if conversation data were ever accessed by unauthorized parties.
Third-party confidential Vendor contracts with non-disclosure clauses, licensed software product keys, supplier pricing agreements Contractual obligations may restrict disclosure of vendor-provided information to third parties. AI services may constitute a third party under those agreements.

10.3 Anonymizing Data Before Sharing

Many legitimate tasks require sharing real data with Claude — log files, configuration snippets, error output. The solution is anonymization: replace sensitive values with realistic-looking placeholders before pasting. Claude works just as effectively with anonymized data.

What to Replace and How

Data Type Original Safe Replacement
Real names John Smith, Dr. Sarah Chen User A, User B (or "a faculty member")
Internal IP addresses 10.42.3.17 10.x.x.x or 192.168.x.x
Credentials password: SecretPass123! password: [REDACTED]
Student IDs B00123456 B00XXXXXX
API keys / tokens sk-ant-api03-abc123... [API_KEY_REDACTED]
Server hostnames db-prod-01.acadiau.ca db-prod-01.example.com
Email addresses jsmith@acadiau.ca user@example.com

[logging]

With the anonymized version, Claude can still help you troubleshoot the configuration structure, identify missing settings, or explain what each parameter does — without any sensitive information leaving your control.

10.4 Always Review Claude's Output

Claude's output is consistently high quality, but high quality is not the same as correct. Claude can confidently produce a command that is wrong for your specific environment, a version number that doesn't match your OS, or a configuration option that was deprecated in the release you're running. Review is not optional — it is how the human-AI partnership stays reliable.

Code and Scripts

The Code Review Protocol

  • Test in a non-production environment first. No Claude-generated script should run on a production server without first being validated on a test system or staging environment.
  • Use safety flags. PowerShell's -WhatIf parameter and bash's --dry-run flags show you what a command would do without actually doing it. Use them.
  • Verify the logic independently. Read through the script and confirm you understand each step. If you can't explain what a line does, research it before running it.
  • Run /security-review before deploying. Any Claude-generated code that will run on a production system should pass a security review first. This catches vulnerabilities that were introduced by Claude's suggestions, not just by your own code.
PowerShell — Test Before You Run 
# Always test destructive operations with -WhatIf first
Get-ChildItem -Path "C:\Logs" -Recurse -Filter "*.log" |
  Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-90) } |
  Remove-Item -WhatIf  # Shows what WOULD be deleted — remove -WhatIf to proceed

# Verify script scope — check what variables and functions are defined
# before running a Claude-generated script in your session
Get-Variable   # See all currently defined variables
Get-Command -CommandType Function  # See all currently defined functions

Documentation

Documentation Review Checklist

  • Verify technical accuracy. Claude may reference commands, paths, or options that are slightly wrong for your specific version. Test every command before including it in a published document.
  • Check version numbers. Confirm that version numbers, build numbers, and software names match the specific software your users are running. Claude's knowledge has a cutoff date and may describe older versions.
  • Verify file paths and URLs. Claude generates plausible paths that may not match your actual environment. Confirm every path against the real system.
  • Subject matter expert review. Before publishing a help article or technical procedure, have someone who performs that task regularly review it for accuracy.

Communications

Communication Review Checklist

  • Add specific details Claude doesn't know. Ticket numbers, contact names, specific dates, internal system names, and escalation paths must be added by you — Claude uses placeholders.
  • Verify the tone. Claude defaults to a professional tone that may be slightly more formal or less formal than Acadia University's communication norms for a given context. Adjust accordingly.
  • Legal and compliance language. Any communication that involves legal obligations, privacy rights, regulatory compliance, or contractual terms should be reviewed by the appropriate office (Privacy Office, Legal Counsel, HR) before sending.
  • Confirm recipients. Before sending a maintenance announcement, confirm the distribution list is complete and correct. Claude doesn't know who is on your mailing lists.

10.5 Security Testing Context

Claude is a capable assistant for security professionals doing authorized security work. It is not a tool for unauthorized access, malware development, or attacks on systems you do not own. This distinction is both ethical and practical — Claude's guidelines restrict it from assisting with certain types of security work regardless of how the request is framed.

Claude Will Help With

  • Penetration testing and vulnerability scanning of your own systems
  • CTF (Capture the Flag) challenge problems
  • Security reviews of your own code (/security-review)
  • Defensive security analysis — hardening configurations, reviewing firewall rules, auditing permissions
  • Explaining how vulnerabilities work conceptually for educational purposes
  • Writing security policy documentation and procedures
  • Analyzing malware behavior from safe, already-captured samples

Claude Will Not Help With

  • Creating malware, ransomware, or weaponized exploit code
  • Attacking systems you do not own or have explicit written authorization to test
  • Generating working exploits for unpatched vulnerabilities in third-party software
  • Bypassing authentication or access controls on systems without documented authorization
  • Surveillance tools designed to operate without the knowledge of the person being monitored
  • Social engineering scripts designed to deceive specific real individuals
Authorized Security Testing at Acadia
If you are conducting authorized security testing of Acadia University systems — such as a scheduled penetration test or vulnerability assessment — document your authorization in writing before starting. Present that authorization context to Claude at the start of your conversation: "I have written authorization from IT management to conduct a penetration test of [system] today. The authorization document is [reference]. Help me with..." This context helps Claude assist effectively with legitimate security work.

10.6 Data Privacy at Acadia University

Acadia University operates under several overlapping privacy frameworks. Technology Services staff working with AI tools need to understand which framework applies to the data they handle and what obligations follow from that.

FERPA

Family Educational Rights and Privacy Act

US federal law protecting student educational records. Applies to Acadia because the university participates in US federal financial aid programs. Restricts disclosure of personally identifiable information from student records. Includes grades, transcripts, enrollment status, financial aid records, and disciplinary records.

AI implication: Never paste student records containing personally identifiable information into Claude without a formal data processing agreement.

PIPEDA

Personal Information Protection and Electronic Documents Act

Canada's federal privacy law governing how private-sector organizations collect, use, and disclose personal information. Requires consent for collection and limits use to the stated purpose. Covers employees, vendors, and anyone whose personal data Acadia holds.

AI implication: Using personal information (employee records, contact details, SINs) in an AI system may constitute a disclosure requiring consent or a data processing agreement.

IT Acceptable Use Policy

Acadia University AUP

Acadia's Acceptable Use Policy governs how staff may use university technology resources, including third-party services accessed through those resources. Review the current policy on the Acadia intranet under Technology Services policies.

AI implication: Using Claude for university work via personal accounts or non-approved tools may not align with AUP requirements. Use the institutionally-sanctioned Claude access.

Enterprise Privacy Agreements
Anthropic's enterprise plans include formal data processing agreements that prevent conversation data from being used for model training and provide commitments about data retention and handling. Contact Technology Services management if your role requires this level of data privacy assurance — for example, if your work regularly involves student records, health data, or employment information that cannot be fully anonymized.
When in Doubt
Before using any AI tool to process personal data about students, employees, or other individuals, consult Acadia's Privacy Office. The cost of a brief consultation is far lower than the cost of a privacy breach.

10.7 Keeping Claude Up to Date

Claude Code, the underlying Claude models, and the skills your team uses all evolve over time. Staying current ensures you have the latest capabilities, security fixes, and model improvements.

Updating Claude Code CLI

Terminal — Update Claude Code CLI

# Verify the installed version
claude --version

# Check if there's a newer version available

10.8 When to Use Claude vs. Escalate

Claude is the right tool for a wide range of IT tasks — but not every task. Part of professional judgment is knowing when to rely on AI assistance and when a situation requires authoritative human expertise, formal processes, or legal guidance.

Situation Use Claude? Rationale
Drafting a script for a known, well-scoped task Yes Claude excels here. Review the output carefully before running in production, but this is exactly the kind of work Claude accelerates.
Debugging a known error message or log entry Yes Claude's training covers thousands of common error conditions. It's faster than documentation search for known issues.
Writing routine IT documentation or help articles Yes Claude produces excellent first drafts. Accuracy review is required before publishing, but the time savings are significant.
Designing a new network architecture or major infrastructure decision Partially Claude provides useful input on options and trade-offs, but major architectural decisions require human expertise, institutional knowledge, and formal review processes.
Analyzing logs and interpreting error patterns Yes One of Claude's strongest use cases. Fast, thorough, and good at identifying patterns across large volumes of log data.
Responding to an active security incident No (or very limited) Active incident response requires authoritative, audited decisions. Use established incident response procedures. Claude can help with reference lookups in parallel, but should not be the primary decision tool.
Processing or analyzing student personal data Only with enterprise agreement FERPA and PIPEDA restrictions apply. Without a formal data processing agreement, student personal data must not be shared with Claude. Consult the Privacy Office.
Making production database changes Generate SQL only Claude can generate and review SQL with high quality. A human must review the generated SQL independently and execute it — never have Claude execute database changes directly in production.
Interpreting legal obligations or contracts No Legal interpretation requires qualified legal counsel. Claude can summarize documents, but contract interpretation, compliance decisions, and legal advice must come from appropriate professionals.
Writing vendor communications or escalations Yes Claude drafts excellent professional communications. Review and add specific details before sending. Useful for organizing the case history before a call.
