6.1 Claude as a Script-Writing Partner

Claude excels at writing the structural and boilerplate code that makes up the bulk of IT automation scripts. You bring the knowledge of your environment — which servers exist, what Active Directory schema you’re using, which edge cases matter — and Claude handles the syntax, error handling patterns, and idiomatic code structure.

The core workflow

Describe what you need
→
Claude drafts the script
→
You review for accuracy
→
Test in lab environment
→
Iterate and deploy

Claude works well for PowerShell, Bash, Python, batch files, and cross-platform automation. The more context you give it — your Windows version, whether AD module is available, which PowerShell version is installed, what the target servers look like — the more accurate the first draft will be.

What Claude does well

Strengths

  • Generating boilerplate (param blocks, try/catch, logging setup)
  • Knowing the right cmdlets, flags, and module names
  • Adding -WhatIf support, -Verbose, and proper pipeline output
  • Writing idiomatic, readable code that follows community conventions
  • Adapting scripts for PowerShell 5.1 vs 7 differences
  • Explaining every line of a script it generates

Where you must add context

  • Acadia-specific server names, OU structures, and naming conventions
  • Which modules are already installed in your environment
  • Service account permissions and what’s available to run as
  • Network topology and reachability constraints
  • Any compliance or audit requirements the script must meet
  • Whether the script runs interactively or as a scheduled task
Critical: Test before production
Never run an untested script on a production server, domain controller, or live database. Always test in a lab environment first. For PowerShell scripts that modify system state, use the -WhatIf flag to preview changes without applying them. For scripts that don’t support -WhatIf, run against a single test target before scaling to a full host list.

6.2 PowerShell Scripts

PowerShell is the primary scripting language for Windows administration at Acadia. Claude is fluent in both PowerShell 5.1 (included with Windows) and PowerShell 7 (cross-platform). Always specify which version you’re targeting.

Example 1 — Inactive AD computer report

Prompt used: “Write a PowerShell script that finds all computers in Active Directory that haven’t logged in for 90 days and exports the results to a CSV. Target PowerShell 5.1 on Windows Server 2022. The ActiveDirectory module is available.”

PowerShell 5.1 Find-InactiveADComputers.ps1 
#Requires -Modules ActiveDirectory
<#
.SYNOPSIS
    Reports Active Directory computers inactive for 90 or more days.
.DESCRIPTION
    Queries AD for computer objects whose LastLogonDate is older than the
    specified threshold and exports results to a timestamped CSV file.
.PARAMETER DaysInactive
    Number of days of inactivity to flag. Default: 90.
.PARAMETER OutputPath
    Directory for the CSV report. Default: current directory.
.EXAMPLE
    .\Find-InactiveADComputers.ps1 -DaysInactive 60 -OutputPath C:\Reports
#>
[CmdletBinding(SupportsShouldProcess)]
param(
    [int]$DaysInactive  = 90,
    [string]$OutputPath = "."
)

Import-Module ActiveDirectory -ErrorAction Stop

$cutoffDate   = (Get-Date).AddDays(-$DaysInactive)
$reportFile   = Join-Path $OutputPath "InactiveComputers_$(Get-Date -Format yyyyMMdd_HHmmss).csv"
$searchBase   = (Get-ADDomain).DistinguishedName

Write-Verbose "Searching for computers inactive since $($cutoffDate.ToShortDateString())"

try {
    $inactiveComputers = Get-ADComputer -Filter { LastLogonDate -lt $cutoffDate } `
        -SearchBase $searchBase `
        -Properties LastLogonDate, OperatingSystem, OperatingSystemVersion, Description, Enabled |
        Select-Object Name, LastLogonDate, OperatingSystem,
                      OperatingSystemVersion, Description, Enabled,
                      @{Name='DaysSinceLogin';Expression={
                          if ($_.LastLogonDate) {
                              [int]((Get-Date) - $_.LastLogonDate).TotalDays
                          } else { 'Never' }
                      }} |
        Sort-Object DaysSinceLogin -Descending
}
catch {
    Write-Error "Failed to query Active Directory: $_"
    exit 1
}

if ($inactiveComputers.Count -eq 0) {
    Write-Host "No computers found inactive for more than $DaysInactive days."
    exit 0
}

if ($PSCmdlet.ShouldProcess($reportFile, "Export $($inactiveComputers.Count) inactive computers")) {
    $inactiveComputers | Export-Csv -Path $reportFile -NoTypeInformation -Encoding UTF8
    Write-Host "Found $($inactiveComputers.Count) inactive computers." -ForegroundColor Yellow
    Write-Host "Report saved to: $reportFile"
}

Example 2 — Disk space monitoring across a server list

Prompt used: “Write a PowerShell script that reads a list of server names from a text file, checks disk space on each one via WMI, and alerts in the console (and in a CSV report) if any drive is more than 80% full. Handle unreachable servers gracefully.”

PowerShell 5.1 Check-DiskSpace.ps1 
[CmdletBinding()]
param(
    [Parameter(Mandatory)]
    [string]$ServerListPath,
    [int]$ThresholdPercent = 80,
    [string]$ReportPath    = ".\DiskSpaceReport_$(Get-Date -Format yyyyMMdd).csv"
)

$servers = Get-Content -Path $ServerListPath | Where-Object { $_ -match '\S' }
$report  = [System.Collections.Generic.List[PSCustomObject]]::new()

foreach ($server in $servers) {
    Write-Verbose "Checking $server..."

    if (-not (Test-Connection -ComputerName $server -Count 1 -Quiet -ErrorAction SilentlyContinue)) {
        Write-Warning "[$server] Unreachable — skipping."
        continue
    }

    try {
        $disks = Get-WmiObject -Class Win32_LogicalDisk `
                     -ComputerName $server `
                     -Filter "DriveType=3" `
                     -ErrorAction Stop
    }
    catch {
        Write-Warning "[$server] WMI query failed: $_"
        continue
    }

    foreach ($disk in $disks) {
        $usedPct = [math]::Round((1 - $disk.FreeSpace / $disk.Size) * 100, 1)
        $entry   = [PSCustomObject]@{
            Server       = $server
            Drive        = $disk.DeviceID
            SizeGB       = [math]::Round($disk.Size / 1GB, 1)
            FreeGB       = [math]::Round($disk.FreeSpace / 1GB, 1)
            UsedPercent  = $usedPct
            Status       = if ($usedPct -ge $ThresholdPercent) { 'ALERT' } else { 'OK' }
        }
        $report.Add($entry)

        if ($usedPct -ge $ThresholdPercent) {
            Write-Warning "ALERT: $server $($disk.DeviceID) is ${usedPct}% full ($([math]::Round($disk.FreeSpace/1GB,1)) GB free)"
        }
    }
}

$report | Export-Csv -Path $ReportPath -NoTypeInformation -Encoding UTF8
Write-Host "`nReport saved to $ReportPath"

$alerts = $report | Where-Object Status -eq 'ALERT'
if ($alerts) {
    Write-Host "`n--- Drives requiring attention ---" -ForegroundColor Red
    $alerts | Format-Table Server, Drive, SizeGB, FreeGB, UsedPercent -AutoSize
}

Tips for better PowerShell output from Claude

What to specifyWhy it mattersExample
PowerShell version 5.1 and 7 have different syntax for some features (ternary operators, null coalescing, pipeline chains) "Target PowerShell 5.1 on Windows Server 2022"
Available modules Claude will only use modules it knows are present — avoiding import failures in production "The ActiveDirectory and ImportExcel modules are installed"
Execution context Interactive scripts vs scheduled tasks have different requirements (no Read-Host, need logging) "This runs as a scheduled task under a service account"
WhatIf requirement Scripts that modify AD, files, or services should be dry-run safe "Add -WhatIf support so I can preview changes before running"
Error handling depth Production scripts need structured error handling with log output, not just Write-Error "Add try/catch around every external call and write errors to a log file"

6.3 Bash / Shell Scripts

Bash scripts are the standard for Linux server administration, WSL environments, and cross-platform automation. Claude generates POSIX-compatible scripts by default and will target a specific shell when asked.

Example — Automated MySQL database backup with retention

Prompt used: “Write a bash script that backs up all MySQL databases to /var/backups/mysql with a timestamp in the filename, and deletes backups older than 7 days. The script will run as a cron job so it should log all output.”

bash mysql-backup.sh 
#!/usr/bin/env bash
# mysql-backup.sh — Full MySQL backup with timestamped files and retention cleanup.
# Intended for cron. Logs to /var/log/mysql-backup.log.
#
# Usage: bash mysql-backup.sh
# Cron:  0 2 * * * /opt/scripts/mysql-backup.sh

set -euo pipefail

readonly BACKUP_DIR="/var/backups/mysql"
readonly LOG_FILE="/var/log/mysql-backup.log"
readonly TIMESTAMP=$(date +%Y%m%d_%H%M%S)
readonly RETENTION_DAYS=7

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"; }

# Ensure backup directory exists
mkdir -p "$BACKUP_DIR"

log "=== MySQL backup started ==="

# Get list of databases (exclude system schemas)
databases=$(mysql --defaults-extra-file=/etc/mysql/backup.cnf \
    -e "SHOW DATABASES;" 2>>"$LOG_FILE" \
    | grep -Ev "^(Database|information_schema|performance_schema|sys)$")

if [[ -z "$databases" ]]; then
    log "ERROR: No databases found or MySQL connection failed."
    exit 1
fi

success_count=0
fail_count=0

for db in $databases; do
    backup_file="$BACKUP_DIR/${db}_${TIMESTAMP}.sql.gz"
    log "Backing up database: $db"

    if mysqldump --defaults-extra-file=/etc/mysql/backup.cnf \
            --single-transaction \
            --routines \
            --triggers \
            "$db" 2>>"$LOG_FILE" | gzip > "$backup_file"; then
        file_size=$(du -sh "$backup_file" | cut -f1)
        log "  OK: $backup_file ($file_size)"
        (( success_count++ ))
    else
        log "  FAILED: $db — see log for details"
        rm -f "$backup_file"   # Remove incomplete file
        (( fail_count++ ))
    fi
done

# Remove backups older than retention period
log "Removing backups older than $RETENTION_DAYS days..."
old_count=$(find "$BACKUP_DIR" -name "*.sql.gz" -mtime +"$RETENTION_DAYS" | wc -l)
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +"$RETENTION_DAYS" -delete
log "Removed $old_count old backup file(s)."

log "=== Backup complete: $success_count succeeded, $fail_count failed ==="

# Exit non-zero if any database failed
[[ "$fail_count" -eq 0 ]]
Credentials in the backup script
The script above uses --defaults-extra-file=/etc/mysql/backup.cnf rather than embedding a username and password in the script. Create /etc/mysql/backup.cnf with chmod 600 permissions. Ask Claude: “What should my backup.cnf file look like?” and it will generate the correct format.

Other common Bash script prompts

  • "Write a bash script that monitors a log file for ERROR lines and sends an email alert when the count exceeds 10 in a minute"
  • "Write a script that checks whether a list of services is running on a remote server via SSH and restarts any that are down"
  • "Create a bash script that syncs a directory to an S3 bucket using the AWS CLI and logs the result"
  • "Write a script that rotates a log file daily, compresses files older than 7 days, and deletes files older than 30 days"

6.4 Python Scripts for Automation

Python is well-suited for data processing, REST API integration, cross-platform automation, and scripts that need complex data structures or libraries. Claude generates well-structured Python with proper argument parsing, logging, and error handling.

Example — AD account audit from a CSV

Prompt used: “Write a Python script that reads a CSV of employee usernames, checks each one against an AD REST API, and outputs a report of any accounts that no longer exist. Use argparse, proper logging, and handle network errors gracefully. Python 3.10+.”

Python 3.10+ audit_ad_accounts.py 
#!/usr/bin/env python3
"""
audit_ad_accounts.py

Reads a CSV of employee usernames, checks each against the AD REST API,
and writes a report of accounts not found in Active Directory.

Usage:
    python audit_ad_accounts.py employees.csv --ad-api https://ad.example.com/api
    python audit_ad_accounts.py employees.csv --username-col samAccountName --output missing.csv
"""

import argparse
import csv
import logging
import sys
import urllib.error
import urllib.request
import json
from datetime import datetime
from pathlib import Path

# ── Logging setup ──────────────────────────────────────────────────────────────
def configure_logging(verbose: bool = False) -> None:
    level = logging.DEBUG if verbose else logging.INFO
    log_file = f"ad_audit_{datetime.now().strftime('%Y%m%d_%H%M%S')}.log"
    logging.basicConfig(
        level=level,
        format="%(asctime)s [%(levelname)-8s] %(message)s",
        handlers=[
            logging.FileHandler(log_file, encoding="utf-8"),
            logging.StreamHandler(sys.stdout),
        ],
    )
    logging.info("Log file: %s", log_file)


# ── AD API helpers ─────────────────────────────────────────────────────────────
def check_account_exists(username: str, api_base: str, timeout: int = 10) -> bool:
    """Return True if the account exists in AD, False if not found or on error."""
    url = f"{api_base.rstrip('/')}/users/{urllib.request.quote(username)}"
    req = urllib.request.Request(url)
    try:
        with urllib.request.urlopen(req, timeout=timeout) as resp:
            return resp.status == 200
    except urllib.error.HTTPError as exc:
        if exc.code == 404:
            return False
        logging.warning("HTTP %s querying %s: %s", exc.code, username, exc.reason)
        return False
    except urllib.error.URLError as exc:
        logging.error("Network error querying %s: %s", username, exc.reason)
        return False


# ── Main ───────────────────────────────────────────────────────────────────────
def main() -> int:
    parser = argparse.ArgumentParser(
        description="Audit AD accounts against a CSV of expected users."
    )
    parser.add_argument("csv_file", help="CSV file containing employee usernames")
    parser.add_argument(
        "--username-col", default="username",
        help="Column name for usernames (default: username)"
    )
    parser.add_argument(
        "--ad-api", default="https://ad.example.com/api",
        help="AD REST API base URL"
    )
    parser.add_argument(
        "--output", default="missing_accounts.csv",
        help="Output CSV for missing accounts (default: missing_accounts.csv)"
    )
    parser.add_argument(
        "--timeout", type=int, default=10,
        help="HTTP request timeout in seconds (default: 10)"
    )
    parser.add_argument("-v", "--verbose", action="store_true", help="Enable debug logging")
    args = parser.parse_args()

    configure_logging(args.verbose)
    log = logging.getLogger(__name__)

    input_path = Path(args.csv_file)
    if not input_path.exists():
        log.error("Input file not found: %s", input_path)
        return 1

    missing_rows: list[dict] = []
    checked = 0
    errors  = 0

    with open(input_path, newline="", encoding="utf-8-sig") as f:
        reader = csv.DictReader(f)

        if args.username_col not in (reader.fieldnames or []):
            log.error(
                "Column '%s' not found. Available columns: %s",
                args.username_col, reader.fieldnames
            )
            return 1

        for row in reader:
            username = row.get(args.username_col, "").strip()
            if not username:
                continue

            checked += 1
            log.debug("Checking: %s", username)

            if check_account_exists(username, args.ad_api, args.timeout):
                log.debug("  Found: %s", username)
            else:
                log.warning("  NOT FOUND in AD: %s", username)
                missing_rows.append(row)

    log.info(
        "Audit complete. Checked: %d | Missing: %d | Errors: %d",
        checked, len(missing_rows), errors
    )

    if missing_rows:
        output_path = Path(args.output)
        with open(output_path, "w", newline="", encoding="utf-8") as f:
            writer = csv.DictWriter(f, fieldnames=missing_rows[0].keys())
            writer.writeheader()
            writer.writerows(missing_rows)
        log.info("Missing accounts written to: %s", output_path)

    return 0 if not missing_rows else 2   # Exit 2 = accounts found missing


if __name__ == "__main__":
    sys.exit(main())
What to notice in this script
Claude structured this with a logging setup function (easy to disable), a separate API helper that handles all error cases without crashing, and a distinct exit code (0 = all good, 1 = script error, 2 = missing accounts found) so it integrates cleanly into monitoring pipelines.

6.5 Debugging Scripts with Claude

When a script fails, Claude can diagnose the error faster than documentation lookups in most cases. The most effective debugging workflow gives Claude the full error message plus the relevant code section.

Step-by-step debugging workflow

  1. Paste the full error message

    Copy the complete error output, including the stack trace or line reference. Partial errors are harder to diagnose. Start with: “I got this error running my PowerShell script:”

  2. Claude identifies the cause

    Claude explains what the error means in plain language, why it’s happening, and what the fix is. For common errors it will also tell you what future variations to watch for.

  3. Add script context if needed

    If Claude’s initial analysis doesn’t fully match your situation, paste the relevant section of the script: “Here’s the section around line 47 that it’s failing on.”

  4. Ask for better error handling

    Once the immediate bug is fixed, ask Claude to harden the script: “Now add try/catch to each external call and write errors to a log file with a timestamp.”

  5. Test the fix

    Apply the fix, re-run with -WhatIf or against a single test target, and confirm the error is gone before running at scale.

Example debugging session

Claude — script debugging session
You

I got this error running my PowerShell script on the management server. It ran fine on my workstation last week:

Import-Module : The specified module 'ActiveDirectory' was not loaded
because no valid module file was found in any module directory.
At C:\Scripts\Find-InactiveADComputers.ps1:2 char:1
+ Import-Module ActiveDirectory
+ ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    + CategoryInfo          : ResourceUnavailable: (ActiveDirectory:String)
    + FullyQualifiedErrorId : Modules_ModuleNotFound,Import-Module
Claude

This error means the ActiveDirectory PowerShell module isn’t installed on the management server. It’s part of the Remote Server Administration Tools (RSAT) and isn’t present by default — your workstation had it installed separately.

Fix depends on the OS:

Windows Server 2016/2019/2022 (install the feature directly):

Install-WindowsFeature -Name RSAT-AD-PowerShell

Windows 10/11 workstation (install as an optional feature):

Add-WindowsCapability -Online `
    -Name Rsat.ActiveDirectory.DS-LDS.Tools~~~~0.0.1.0

After installing, close and reopen PowerShell — the module won’t be available in the same session. The module connects to AD remotely, so it doesn’t need to run on the DC itself; any machine on the domain with RSAT installed will work.

If you don’t have permission to install features on that server, ask your sysadmin to run the Install-WindowsFeature command for you.

You
That fixed it, but now I’m getting "Access is denied" when it tries to query AD. The script runs as my regular account.
Claude

The Get-ADComputer cmdlet needs read access to Active Directory. Regular user accounts can usually run basic AD queries, but "Access is denied" at the script level typically means one of two things:

  1. Your account doesn’t have permission to read the specific OU you’re searching
  2. The -SearchBase OU path is incorrect and AD is refusing the query with an ambiguous error

Quick diagnostic — run this interactively under your own account to confirm you can reach AD at all:

Get-ADComputer -Filter * -ResultSetSize 1

If that works, the issue is the SearchBase. Run (Get-ADDomain).DistinguishedName to get the correct root DN to pass to the script.

Common script errors Claude can diagnose instantly

Error typeTypical causeWhat to give Claude
Module not found RSAT, ImportExcel, or other module not installed on target machine Full error + OS version + where the script runs
Access denied Script running as wrong account, or OU permissions insufficient Error + the cmdlet that failed + service account name
WMI / CIM query failures Firewall blocking WMI ports, or WMI service not running on target Error message + target server name + network topology
Encoding errors in CSV Non-UTF8 data or BOM issues between PowerShell versions Error + what the CSV contains + how it was created
Pipeline object type mismatches Cmdlet output type changed between PowerShell 5.1 and 7 Error + the specific pipeline expression that fails

6.6 Running Scripts Through Claude

In some interfaces, Claude can write and execute scripts in a single workflow — you describe what you need, Claude generates the script, and you can ask it to run the script and interpret the results.

From the Terminal CLI

When Claude has Bash permission in the CLI, it can:

  • Write a script file (backup.sh, CheckAccounts.ps1) directly to disk
  • Execute the script and capture its output
  • Display the output alongside an interpretation of what happened
  • Iterate based on the results — if an error occurs, Claude reads the output and suggests a fix without you having to re-paste anything
Permission and confirmation
Claude will ask for explicit confirmation before running any script that modifies system state. For read-only scripts (reporting, auditing, querying), Claude can run them without interrupting you if you’ve granted the appropriate Bash permission. See Module 4 — Permissions for how to set this up correctly.

From VS Code

With Claude active in VS Code, the write-test-iterate loop becomes tightly integrated:

  1. Ask Claude to write the script — it appears directly in your editor
  2. Claude opens the integrated terminal and runs the script
  3. Terminal output appears in the VS Code panel and Claude sees it
  4. You can ask: “The output shows three servers failed — what went wrong?”
  5. Claude reads the output and responds in the same conversation

What Claude will not do without explicit confirmation

Always asks first

  • Delete files or directories
  • Modify or disable AD accounts
  • Change system configuration or registry values
  • Restart services or servers
  • Write to locations outside the current project directory
  • Run scripts that would affect more than the current session

Can run without prompting (if permitted)

  • Query AD for computer or user objects
  • Check disk space, service status, or network connectivity
  • Read log files and generate reports
  • Run scripts with -WhatIf flag active
  • List directory contents or file metadata
  • Query databases in read-only mode

6.7 Script Writing Best Practices

Whether Claude generates the first draft or you write it yourself, every production script at Acadia Technology Services should follow these practices. When prompting Claude, you can ask it to enforce all of them in a single request: “Write this script with full error handling, logging, -WhatIf support, and parameter validation.”

🧪

Start with dry-run mode

Add -WhatIf to PowerShell scripts and --dry-run or an -n flag to Bash/Python scripts. Run with the flag before the first real execution.

📋

Always write a log file

Automated scripts run unattended. A log file with timestamps is your only record when something goes wrong at 2 am. Use Start-Transcript in PowerShell or redirect stdout in bash.

⚠️

Handle errors explicitly

Every call to an external system (AD, WMI, database, API) can fail. Wrap each in try/catch and log the exception. A script that silently continues after an error produces incorrect results.

🔧

Use parameters, not hardcoded values

Replace hardcoded server names, paths, and thresholds with parameters (-Server, -Path, -Threshold). This makes scripts reusable across environments without editing source.

🚫

Test with non-production data

Never test a script that modifies data on a live dataset. Use a test OU in AD, a staging database, or a non-critical server. Validate with -WhatIf first, then a single real target.

📂

Version-control your scripts

Every script that runs in production should live in a git repository. Include a changelog comment block at the top, increment a version number with each change, and commit after every working change.

Asking Claude to enforce best practices on an existing script

If you have a working but rough script that lacks these practices, paste it into Claude and ask:

Credentials and secrets
Never hardcode passwords, API keys, or service account credentials in a script. Claude will remind you of this and suggest the correct alternative: PowerShell’s Get-Credential, a .cnf file with restricted permissions, environment variables, or a secrets manager. If you paste a script that contains credentials, Claude will flag them immediately and suggest how to remove them safely.

Module 6 Summary

Claude is a capable co-author for IT automation scripts in PowerShell, Bash, and Python. The most effective workflow is: describe your environment and requirements clearly, let Claude generate a structured first draft, then review carefully and test in a safe environment before production use. Claude will enforce good practices — error handling, logging, parameter validation — if you ask, or you can add them after the fact with a targeted follow-up prompt.

The core rule for all script work with Claude: Claude writes, you validate, you approve. Claude does not know your production environment, your AD schema, or the edge cases that matter to Acadia IT operations — you do. Use Claude to handle the syntax and structure, and bring your own expertise to the logic and validation.
