3.1 What Are Skills?

Skills are reusable instruction files that extend Claude's behavior for specific tasks. Instead of repeating the same context or instructions every time you start a conversation, you package them once into a skill file — and invoke it with a single slash command whenever you need it.

📄

Markdown Files

Skills are plain .md files with YAML frontmatter. No special syntax to learn.

⚡

Slash Commands

Invoke any skill by typing /skill-name in the Claude chat interface.

🔄

Reusable Workflows

Think of skills as macros or templates — define once, use many times across projects.

When you type /skill-name, Claude reads that skill file before responding, giving it specialized knowledge or a step-by-step workflow to follow. Skills are version-controllable, shareable with your team, and composable — you can build complex workflows from simple skill building blocks.

Anatomy of a Skill File

Every skill is a Markdown file with a YAML frontmatter block at the top, followed by instructions for Claude in plain prose or structured Markdown.

.claude/skills/my-skill/SKILL.md Markdown 
---
name: my-skill
description: Brief description of when this skill should fire
---

# Instructions for Claude

When this skill is invoked, do the following:

1. First, read the relevant files in the project
2. Apply the specified transformation or analysis
3. Return results in the format described below

## Output Format

- Use bullet points for findings
- Highlight critical issues in bold
- Include a summary section at the end

The frontmatter block (between the --- delimiters) contains metadata Claude uses to identify and trigger the skill. The body of the file is freeform Markdown — write it like you would write instructions to a capable colleague.

Key Concept
Description accuracy matters. The description field in the frontmatter tells Claude when this skill applies. Claude reads skill descriptions to decide whether to suggest a skill automatically. Write descriptions that clearly state the trigger condition.

3.2 Where Skills Live

Claude Code looks for skills in two locations, each with a different scope. Understanding the difference helps you decide where to put each skill.

🌎 Global Skills

Path: C:\Users\USERNAME\.claude\skills\

Available in every project on your machine. Use this location for general-purpose skills you want everywhere: your standard code review style, your department's file naming convention, your personal preferences.

📁 Project Skills

Path: [Project Root]\.claude\skills\

Only available when working inside that specific project. Use this location for project-specific workflows: an incident report template for your helpdesk repo, a deployment checklist for a specific app.

When both a global and a project-level skill share the same name, the project-level skill takes precedence. This lets teams override global defaults with project-specific behaviour.

Directory Structure

Skill directory layout (Windows) Directory Tree 
C:\Users\USERNAME\
└── .claude\
    └── skills\
        ├── code-review\          <-- invoked as /code-review
        │   └── SKILL.md
        ├── theme\                <-- invoked as /theme
        │   └── SKILL.md
        └── file-naming\          <-- invoked as /file-naming
            └── SKILL.md

C:\Projects\helpdesk-portal\      <-- your project root
└── .claude\
    └── skills\
        └── incident-report\      <-- invoked as /incident-report
            └── SKILL.md          <-- only available in this project
Naming Convention
The folder name inside skills\ is the invoke name. A folder called incident-report becomes /incident-report. Keep folder names lowercase with hyphens — no spaces, no underscores.

3.3 How to Invoke a Skill

Invoking a skill is as simple as typing a slash followed by the skill name in the Claude chat input. You can also pass additional context or arguments after the skill name.

  • Type /skill-name and press Enter to invoke with no arguments
  • Type /skill-name some extra context to pass arguments Claude will incorporate
  • Tab completion works in both the Claude Desktop App and the CLI REPL — press Tab after the slash to see available skills
  • Skills are case-insensitive: /Code-Review and /code-review both work

Live Example — Invoking /code-review

The following mockup shows what happens when you invoke the /code-review skill in a project with uncommitted changes:

Claude Code — helpdesk-portal
You
/code-review focus on error handling
Claude
Code Review — helpdesk-portal

I've reviewed your current diff (3 files changed, 87 insertions, 12 deletions). Here are my findings, with particular attention to error handling as requested.

🔴 Issues Found

1. Unhandled promise rejection — src/api/tickets.js:42
The fetchTicket() call has no .catch() handler. If the API is unreachable, this will produce an unhandled rejection and silently fail in production.

2. Missing null check — src/components/TicketDetail.jsx:18
ticket.assignee.name will throw if assignee is null (unassigned tickets). Use optional chaining: ticket.assignee?.name.

🟢 Looks Good

Authentication middleware and the new retry logic in src/utils/httpClient.js are well-structured. The exponential backoff implementation is correct.

Summary: 2 issues to fix before merging. No security concerns. Test coverage looks adequate for the changed paths.
Pro Tip
You can pass any free-text argument after the skill name. /code-review focus on error handling narrows the review scope. /code-review --comment posts findings as inline GitHub PR comments. Check each skill's documentation for supported arguments.

3.4 Skills Available for Technology Services

The following skills are either included with Claude Code or have been configured for Acadia Technology Services staff. They are available globally (once installed to your ~\.claude\skills\ directory) or on a per-project basis.

Skill Invoke What It Does Scope
theme /theme Applies the Acadia University visual theme to any deliverable — HTML pages, Word documents, PowerPoint presentations, or dashboards. Ensures colours, fonts, and layout conform to brand standards. Global
file-naming /file-naming Names final deliverables using the department convention: DEPARTMENT - Short Name - YYYYMMDD. Ensures consistent file naming across Technology Services outputs. Global
code-review /code-review Reviews the current git diff for correctness bugs, logic errors, and improvement opportunities. Supports effort levels (low / medium / high / ultra) and the --comment flag to post findings as inline PR comments. Global
security-review /security-review Reviews code changes specifically for security vulnerabilities: injection risks, authentication flaws, secrets in code, insecure dependencies, and improper access control. Global
simplify /simplify Refactors changed code for clarity and efficiency. Looks for duplication, unnecessary complexity, and opportunities to use standard library features. Applies fixes directly to the working tree. Global
update-config /update-config Configures Claude Code settings and hooks via natural language. Tell it what you want ("allow npm commands", "run linting after file writes") and it generates the correct settings.json changes. Global
init /init Scans the current project and generates a CLAUDE.md file documenting the codebase — structure, dependencies, key patterns, and how to build and run the project. Ideal for onboarding. Global
incident-report /incident-report Guides staff through generating a structured IT incident report. Asks for date/time, systems affected, description, impact, resolution steps, and follow-up actions, then produces a formatted report. Project
Note
Skills marked Global are available in every project once installed to your user-level skills directory. Skills marked Project are installed in a specific project's .claude/skills/ folder and are only available there.

3.5 Installing Skills

Installing a skill is simply a matter of placing the skill's .md file in the correct directory. No compilation, no package manager, no restart required beyond refreshing your Claude session.

Installing a Global Skill (Available Everywhere)

  1. Obtain the skill file. Download the .md file from your team's shared repository, or create one from scratch. The file should contain YAML frontmatter and instructions for Claude.

  2. Navigate to your global skills directory. Open File Explorer or a terminal and go to:

    C:\Users\USERNAME\.claude\skills\

    If the skills folder does not exist yet, create it.

  3. Create a subfolder for the skill. The folder name becomes the invoke name. For a skill you want to call as /incident-report, create a folder named incident-report.

    mkdir C:\Users\USERNAME\.claude\skills\incident-report

  4. Place the skill file inside the folder. The file must be named SKILL.md (all caps).

    C:\Users\USERNAME\.claude\skills\incident-report\SKILL.md

  5. Refresh your Claude session. Close and reopen the Claude Desktop App, or type /reload in the CLI REPL. The new skill will appear in tab-completion.

  6. Test the skill. Type /incident-report in the chat to verify the skill loads and Claude responds as expected.

Installing a Project-Level Skill

The process is identical, but the target directory is inside your project root instead of your user home directory:

Project-level skill installation PowerShell 
# Navigate to your project root
cd C:\Projects\helpdesk-portal

# Create the skills directory if it doesn't exist
mkdir .claude\skills\incident-report -Force

# Copy your skill file into place
Copy-Item .\downloads\incident-report.md .\.claude\skills\incident-report\SKILL.md
Source Control
Project-level skill files in .claude/skills/ should be committed to your git repository. This lets your whole team share the same skills automatically when they clone the project. Consider adding a README.md to the .claude/skills/ folder explaining your project's custom skills.

3.6 Creating a Custom Skill

Writing a skill is one of the most valuable things you can do to improve your team's productivity with Claude. Any repeated workflow, standard document format, or specialized analysis your team performs regularly is a good candidate for a skill.

Example: The Incident Report Skill

Acadia Technology Services staff regularly produce IT incident reports. The following skill guides Claude to collect all required information and produce a consistently formatted report — eliminating the need to remember the required fields or format each time.

.claude/skills/incident-report/SKILL.md Markdown 
---
name: incident-report
description: Generate a structured IT incident report from a brief description of what happened
---

# Incident Report Generator

When invoked, ask the user for the following information, one prompt at a time
if not already provided:

1. Date and time of the incident
2. Systems affected (servers, applications, network segments, services)
3. Description of the problem (what happened, how it was first noticed)
4. Impact on users (number affected, severity of disruption, duration)
5. Steps taken to resolve (timeline of actions, who was involved)
6. Current status and follow-up actions required

Then produce a formatted incident report following Acadia Technology Services
standards, using the structure below.

## Output Format

# IT Incident Report — [Systems Affected]
**Date/Time:** [date and time]
**Status:** [Resolved / Ongoing / Monitoring]
**Reported by:** [if provided]

## Incident Summary
[2-3 sentence description of what happened]

## Systems Affected
[Bullet list of affected systems]

## User Impact
[Description of impact, including number of users affected and duration]

## Timeline of Events
| Time | Action |
|------|--------|
| HH:MM | Event description |

## Root Cause
[Root cause if identified, or "Under investigation" if not]

## Resolution
[Steps taken to resolve the incident]

## Follow-up Actions
- [ ] Action item 1
- [ ] Action item 2

---
*Report generated by Acadia Technology Services*

Invoking the Skill

Once installed, a staff member simply types /incident-report in Claude. Claude will then walk through the required questions and produce the formatted report:

Claude Code — helpdesk-portal
You
/incident-report The email server went down this morning and we couldn't receive mail for about two hours.
Claude
I'll help you create a structured incident report. I have a description of the problem — let me gather the remaining required details.

1. Date and time: When exactly did the incident begin? (e.g., 2026-06-28 at 08:15 AST)

2. Systems affected: Which email server or service was impacted? (e.g., Exchange server, Office 365, specific hostname)
You
Started at 07:45 AST. It was the on-prem Exchange server, EXCAS01.
Claude
Thank you. A few more questions:

3. User impact: Approximately how many users were affected? Were they unable to send, receive, or both?

4. Steps taken to resolve: What actions were taken, and who was involved?

After collecting all information, Claude generates the complete formatted report ready for your incident management system or team documentation.

Design Principles for Effective Skills

  • One skill, one job. A skill that tries to do too much becomes hard to invoke reliably. Split complex workflows into smaller, focused skills.
  • Write for Claude, not for humans. The skill body is instructions for Claude — be direct and specific. "Ask the user for X, then Y" works better than "the user may provide X or Y."
  • Include an output format. Specify exactly how you want the output structured. Claude will follow precise format descriptions faithfully.
  • Test with edge cases. Try invoking the skill with minimal input, maximal input, and unusual inputs to ensure it handles all situations gracefully.

3.7 Tips for Using Skills Effectively

Skills Stack — Use Them Together

You are not limited to one skill per conversation. Invoke a skill, get a result, then invoke another. The conversation context carries forward between invocations.

Workflow Example
  1. Use /code-review to identify issues in a pull request
  2. Ask Claude to fix the identified issues
  3. Use /security-review to check the same diff for vulnerabilities
  4. Use /simplify to refactor any complex sections
Each skill adds its perspective to the same conversation — Claude retains full context throughout.

Pass Context After the Command

Most skills accept free-text arguments after the command name. Claude incorporates this context when running the skill:

Invocation Effect
/code-review Full review of the current diff
/code-review focus on error handling Review emphasising error handling patterns
/code-review --comment Posts findings as inline GitHub PR comments
/code-review --fix Applies suggested fixes directly to the working tree
/theme apply to the dashboard only Applies the Acadia theme to a specific section
/incident-report brief description of outage here Pre-populates the problem description and skips that question

Check Available Skills with /help

To see all skills currently available in your session, type /help in the Claude chat. The response will list every registered skill with its name and description — useful when working in a new project or after adding new skills.

Keep Descriptions Accurate
Claude reads skill description fields to decide when to proactively suggest a skill. If a description is too vague or inaccurate, Claude may suggest the skill at the wrong time — or never. When you create or edit a skill, test that the description correctly captures the trigger condition: "when should this skill be used?"

Share Skills with Your Team

Project-level skills committed to your git repository are automatically available to every team member who clones the repo. This is the recommended way to distribute Technology Services standard workflows:

  1. Create the skill in .claude/skills/skill-name/SKILL.md
  2. Commit and push to the main branch
  3. Team members pull the changes — the skill is immediately available
  4. Iterate on the skill by editing the SKILL.md file and committing updates
