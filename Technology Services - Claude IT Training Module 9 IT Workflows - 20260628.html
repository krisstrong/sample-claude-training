9.1 Claude in the IT Day-to-Day

Technology Services staff handle a high volume of mixed work every day — troubleshooting tickets, writing documentation, reviewing code, drafting communications, and managing change. Claude does not replace the judgement that makes IT professionals effective. It handles the mechanical, time-consuming work so that judgement can be applied where it matters most.

Think of Claude as a skilled assistant who is always available, never loses patience with repetitive tasks, and can instantly recall documentation for thousands of technologies — but who always needs a human to review the result before it becomes real. Claude can search for the right command, draft the explanation, and interpret the error message. You decide whether the output is correct and safe to act on.

🔍

📋

Draft the Communication

Maintenance windows, vendor escalations, plain-English explanations — Claude drafts in seconds.

📊

Read the Logs

Paste a wall of log output and ask Claude what went wrong. It reads faster than any human.

🛠

Interpret Errors

Unknown error code? Paste it and let Claude explain the cause and likely fix.

📝

Summarize Tickets

Long email chains and incident threads distilled into a clear, structured summary.

📄

Plan the Change

Change requests, pre-flight checklists, and rollback procedures written systematically.

Guiding Principle
Claude handles the mechanical work. You provide the context, judgement, and authorization. The combination is far more effective than either alone.

9.2 Troubleshooting Assistance

When a user reports an issue, Claude can be a first-line reference before you even pick up the phone. Describe the symptom, the affected system, and any recent changes, and Claude will suggest a structured troubleshooting path — pointing you to the right commands, settings, and logs to check.

Example: Network Drive Access After Windows Update

Claude — Troubleshooting Session
You
A user reports they can't connect to network drives after Windows Update KB5034441. What should I check?
Claude

KB5034441 (January 2024 security update) is known to affect SMB authentication in some environments. Here's a structured troubleshooting path:

1. Verify Group Policy Application
Run gpresult /r on the affected machine and look for any drive-mapping policies that may now be blocked. Check for errors in the output.

2. Check DNS Resolution
Run nslookup fileserver.acadiau.ca and confirm it resolves correctly. A DNS failure will look like a network drive issue. Also try ping fileserver.acadiau.ca -4.

3. Verify SMB Settings
Run Get-SmbClientConfiguration | Select-Object EnableSecuritySignature, RequireSecuritySignature — this update enforces SMB signing more aggressively. If the server doesn't support signing, the connection will fail.

4. Test Manual Connection
Try net use Z: \\fileserver\share /user:domain\username from a command prompt. The error message returned is more specific than the generic Windows error dialog.

5. Check Event Logs
Run Get-EventLog -LogName System -Source "srv" -Newest 20 on the file server to look for SMB session or authentication errors around the time of failure.

Most likely cause: SMB signing mismatch. KB5034441 changed the default SMB signing behavior. Confirm server-side signing settings match the new client requirements.

The Iterative Troubleshooting Loop

Troubleshooting with Claude is an iterative cycle. Start with the symptom, run the suggested diagnostics, and paste the output back for interpretation. Claude adjusts its analysis based on what you find.

  1. Describe the symptom — include the error message, affected system, OS version, and any recent changes (updates, config changes, new deployments).

  2. Claude identifies likely causes — it cross-references the symptom against known issues and suggests diagnostic commands ranked by likelihood.

  3. Run the diagnostics — execute the suggested commands on the affected system.

  4. Paste the output back — Claude interprets the results and narrows the diagnosis.

  5. Repeat until resolved — each round gives Claude more information to work with.

Example: Interpreting netstat Output

Claude — Network Diagnostics
You
I ran netstat -ano on the problematic server. Here's the output: [paste]. Are there any unusual connections I should be concerned about?
Claude

Looking at the netstat output, a few things stand out worth investigating:

Connections I'd flag for review:

  • Any ESTABLISHED connections on port 445 (SMB) from unexpected external IP ranges — SMB should only be internal
  • Processes with PID 0 listed as listening — these can indicate kernel-level listeners
  • High volumes of TIME_WAIT states that suggest connection churn or a misbehaving application

To identify the process behind a specific PID, run:
Get-Process -Id [PID] | Select-Object Name, Path, StartTime

If you share the actual output, I can identify specific lines that warrant attention.

9.3 Log Analysis

Log files are one of the richest sources of diagnostic information in IT — and one of the most time-consuming to read. Claude can process hundreds of log lines in seconds, identify anomalies, correlate events, and summarize findings in plain English.

Collecting Logs for Claude

The simplest approach is to export log data to a text format and paste it into the conversation. For large logs, extract the relevant time window or most recent entries.

PowerShell — Export Windows Event Log to CSV 
# Export the last 200 Application log events to CSV
Get-EventLog -LogName Application -Newest 200 |
  Select-Object TimeGenerated, EntryType, Source, EventID, Message |
  Export-Csv -Path "$env:DESKTOP\events.csv" -NoTypeInformation

# Then open the CSV, copy the relevant rows, and paste into Claude
PowerShell — Extract System Errors from the Last 24 Hours 
# Get only Error and Warning entries from the past day
$since = (Get-Date).AddHours(-24)
Get-EventLog -LogName System -EntryType Error,Warning -After $since |
  Select-Object TimeGenerated, EntryType, Source, EventID, Message |
  Format-List | Out-String | Set-Clipboard

# Output is now in your clipboard — paste directly into Claude

Asking Claude to Analyze Logs

Claude — IIS Log Analysis
You
Here are the last 100 lines of our IIS log from the Acadia portal server. Are there any 4xx or 5xx errors? What's causing them?

[paste of log lines]
Claude

From the IIS log you shared, I can see the following HTTP error patterns:

404 Not Found — 23 occurrences
All requests for /wp-admin/ and /wp-login.php — these are automated WordPress scanning attempts hitting a non-WordPress site. Expected noise; no action required unless you want to add a firewall rule to block the source IPs.

500 Internal Server Error — 4 occurrences
All four errors hit /api/v2/reports/generate at 14:23, 14:31, 14:44, and 14:52. The same request pattern repeating at irregular intervals suggests a background job or scheduled task that's failing. Check the application event log on the server around those timestamps for the underlying exception.

503 Service Unavailable — 1 occurrence
Single instance at 14:19, before the 500 errors started. A brief application pool crash followed by recovery would explain this sequence.

Recommendation: Focus on the application event log around 14:19 for the root cause. The 500 errors are likely a retry loop failing after the initial pool crash.

Firewall Log Analysis

PowerShell — Extract Windows Firewall Drop Events 
# Get firewall drop events from the Security log
Get-WinEvent -FilterHashtable @{
  LogName   = 'Security'
  Id        = 5152  # Filtering Platform Packet Drop
  StartTime = (Get-Date).AddHours(-6)
} | Select-Object TimeCreated,
  @{N='SourceIP';E={$_.Properties[3].Value}},
  @{N='DestPort';E={$_.Properties[11].Value}},
  @{N='Protocol';E={$_.Properties[7].Value}} |
  Group-Object SourceIP |
  Sort-Object Count -Descending |
  Select-Object -First 20 |
  Format-Table -AutoSize

Paste the output to Claude with the prompt: "Identify any IP addresses making unusually high connection attempts. Flag anything that looks like a port scan or brute-force pattern."

Before You Share Logs
Review logs for sensitive data before pasting. Remove or redact real IP addresses of internal hosts, usernames, and any data that could identify individuals. See Module 10 for anonymization techniques.

9.4 Security Reviews with /security-review

The /security-review skill audits code changes against common vulnerability patterns before they reach production. Any time you write or modify a script, configuration file, or application code, running a security review is a fast and low-effort safety check.

How to Run a Security Review

  1. Make your code changes on a branch — /security-review works by examining the current diff, so your changes need to be staged or committed.

  2. Type /security-review in Claude Code — no additional arguments needed for a standard review. Claude reads the diff automatically.

  3. Claude analyzes the diff — it checks against OWASP Top 10 vulnerability categories, looks for exposed credentials, injection risks, insecure patterns, and missing validation.

  4. Review the findings report — findings are prioritized as Critical, High, Medium, or Low. Each finding includes the file, line, the issue, and a recommended fix.

  5. Address Critical and High findings before merging — Medium and Low findings should be documented for follow-up. Never merge code with unaddressed Critical findings.

What /security-review Catches

Common High-Severity Findings

  • Hardcoded credentials — passwords, API keys, or connection strings written directly in code
  • SQL injection — string concatenation used to build SQL queries instead of parameterized statements
  • Command injection — unvalidated user input passed to shell commands or Invoke-Expression
  • Missing authentication — endpoints or functions that skip authorization checks

Common Medium-Severity Findings

  • Insecure file permissions — scripts that create files world-writable or leave sensitive files readable by all users
  • Sensitive data in logs — logging statements that output passwords, tokens, or PII
  • Weak cryptography — use of MD5, SHA1, or DES where stronger algorithms should be used
  • Missing input validation — functions that accept user input without length, type, or format checks
Best Practice
Run /security-review on every script you write with Claude's help before deploying it to production. Claude-generated code is generally good but not infallible — the security review adds a second pass with different prompting specifically focused on vulnerabilities.

9.5 Ticket and Issue Summarization

IT ticketing systems accumulate dense text — long email threads, verbose incident notes, multi-day comment chains. Claude converts this raw material into structured, actionable summaries in seconds.

Effective Summarization Prompts

Goal Prompt to Use
Incident summary "Here is a thread of 15 emails about a network outage. Summarize what happened, what the root cause was, and what the resolution was."
Formal post-mortem "Convert this incident description into a formal post-mortem document with sections for: incident summary, timeline, root cause, contributing factors, and action items."
Vendor escalation "I need to escalate this ticket to the vendor. Write a professional summary of the issue with all the technical details from these notes, including dates, error codes, and steps already taken: [paste]"
Ticket categorization "Categorize these 50 helpdesk tickets into themes. What are the top 3 issue types and how many tickets fall into each?"
Status update "Based on these investigation notes, write a brief status update for the affected department explaining what we know and what we're doing next."

Example: Incident Summarization

Claude — Incident Summary
You
Here is the full email thread from the file server outage last Tuesday. Summarize what happened, the root cause, and the resolution. Format it as a post-mortem I can put in the ticketing system.

[paste of email thread]
Claude

Incident Post-Mortem — File Server Outage
Incident Date: [Date from thread]  |  Duration: [X hours]  |  Severity: High

Summary
The primary file server (FS01) became unavailable to all users from approximately [start time] to [end time], preventing access to mapped network drives university-wide.

Root Cause
A storage volume ran out of free space due to a backup job that had been writing to the wrong target path for three days. The volume reaching 100% capacity caused the SMB service to stop accepting new connections.

Resolution
The erroneous backup files were moved to the correct destination volume, freeing 180 GB of space. SMB service was restarted, and drive mappings reconnected automatically for all users within 15 minutes.

Action Items

  • Configure disk-space alerts on all file server volumes (threshold: 80%)
  • Audit backup job targets to confirm correct paths are configured
  • Document runbook for SMB service restart procedure

9.6 Writing Professional IT Communications

Drafting communications is often the slowest part of IT work — not because the content is difficult, but because finding the right words for mixed audiences takes time. Claude produces professional first drafts for maintenance windows, vendor escalations, and plain-English explanations far faster than writing from scratch.

Communication Prompts That Work Well

Maintenance Window Announcement

Tell Claude the system, the window, and the impact:

"Write a maintenance window announcement for users. We're patching the file server this Saturday from 11 PM to 2 AM. Network drives will be unavailable during this window. The audience is all Acadia staff."

Vendor Escalation

Give Claude the issue history and let it structure the case:

"Draft a professional email to [vendor] about a recurring issue with their SSO integration that's been impacting faculty login for three weeks. I've attached our case notes."

Non-Technical Explanations

Claude excels at translating technical concepts for general audiences:

"Write a plain-English explanation of this SSL certificate error to send to a non-technical department head who needs to understand why their website is showing a security warning."

Service Disruption Notices

For unplanned outages, Claude drafts the user-facing notice:

"Write a brief service disruption notice for the Acadia portal. The issue is a database connection error causing login failures. We're investigating and estimate 30-60 minutes to resolve."

Always Review Before Sending
Claude-drafted communications are first drafts. Always review before sending. Add specific names, ticket numbers, contact information, and any details Claude would not know. Confirm the tone matches Acadia University's communication standards, and verify any legal or compliance-related language with the appropriate office before distribution.

9.7 Change Management and Planning

Systematic change management reduces risk and improves outcomes. Claude helps create the documentation that makes changes go smoothly — not by replacing the expertise required to plan a change, but by ensuring the documentation is thorough and nothing is overlooked.

Change Planning Prompts

Task Example Prompt
Change request document "Write a change request document for migrating our file server from Windows Server 2016 to 2022. Include: summary, business justification, scope, risk assessment, rollback plan, and approval requirements."
Pre-flight checklist "Create a pre-flight checklist for deploying a Windows cumulative update to production servers. Include verification steps, backup confirmation, and communication steps."
Rollback procedure "What are the rollback steps if this Active Directory schema change causes issues? Include how to verify the rollback was successful."
Stakeholder identification "List all the stakeholders we should notify before a network maintenance window affecting the main campus building. Include suggested lead times for each group."
Risk assessment "Identify the risks in this server migration plan and suggest mitigations for each. Rate each risk as High, Medium, or Low."
Claude Plans, Humans Authorize
Change documentation produced by Claude is a starting point. It must be reviewed by the staff member responsible for the change and approved through Acadia Technology Services' change management process. Claude's plan may omit environment-specific steps that only you know about.

9.8 Writing User Help Articles

Technology Services produces help content for the Acadia community — setup guides, how-to articles, and troubleshooting instructions for the wide range of systems students, faculty, and staff use. Claude can produce polished first drafts that are structurally sound and clearly written, significantly reducing the time from idea to published article.

Help Article Workflow

  1. Describe the task — tell Claude what the article should cover: "Explain how to set up Duo two-factor authentication on a new phone."

  2. Specify the audience — "Write this for students who may not be technical. Avoid jargon and explain each step in plain language."

  3. Claude drafts the article — a structured document with a clear title, introduction, numbered steps, and a troubleshooting section.

  4. Review for accuracy — Claude does not know your specific Duo configuration, portal URL, or institution-specific steps. Verify every step against the actual system.

  5. Apply theme and naming — use /theme to apply the Acadia visual theme if publishing as HTML, and /file-naming to generate the correct filename for the Technology Services file naming standard.

Prompts for Different Article Types

Setup Guide

"Write a step-by-step guide for connecting to the Acadia VPN using Cisco AnyConnect on Windows 11. The audience is new students. Include screenshots placeholders and a common errors section."

Troubleshooting Article

"Write a troubleshooting article for students who can't log in to Brightspace. Structure it as: check these things first, common causes and fixes, and when to contact IT."

Policy Explainer

"Write a plain-language explanation of Acadia's acceptable use policy for AI tools. The audience is faculty. Tone should be collegial and informative, not legalistic."

Quick Reference

"Create a one-page quick reference card for Acadia email on mobile devices. Cover iOS and Android setup. Use a two-column layout with screenshots placeholders."

Time Savings in Practice
A well-prompted Claude draft of a help article typically takes 2–3 minutes to generate and 10–15 minutes to review and correct for accuracy. Writing the same article from scratch typically takes 45–90 minutes. The time savings are real — but accuracy review cannot be skipped.
