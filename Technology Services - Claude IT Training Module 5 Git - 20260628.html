5.1 Why Use Claude for Git Work?

Git is the foundation of collaborative software and script development at Acadia. Claude understands git’s entire workflow — from staging individual files through branching strategies, pull requests, and merge conflict resolution — and can act as a knowledgeable collaborator at every stage.

What Claude can do with git

  • Write clear, well-structured commit messages
  • Draft pull request descriptions that explain why changes were made
  • Review your staged diff and flag bugs or improvements
  • Interpret git log, git blame, and git show output
  • Guide you through merge conflict resolution step by step
  • Generate .gitignore files, hooks, and repository configuration
  • Explain what any git command or flag does in plain language

Where this saves time

  • No more staring at a blank commit message field
  • Catch bugs in scripts before they reach production
  • Understand a repository you’ve just inherited
  • Decode cryptic merge conflicts without manual line-by-line comparison
  • Onboard teammates by generating repository documentation from history
  • Audit what changed between releases without reading every commit
Built-in safety
Claude never skips git hooks (--no-verify) unless you explicitly ask it to. This keeps your pre-commit checks, linting passes, and secret-scanning hooks intact. If a hook fails, Claude investigates the underlying problem rather than bypassing the check.

5.2 Writing Commit Messages

A good commit message communicates why a change was made, not just what changed. Claude follows established conventions automatically:

  • Imperative mood — "Fix token expiry check" not "Fixed token expiry check"
  • Subject line under 72 characters — keeps it readable in terminals and GitHub
  • Blank line between subject and body — required by many git tools
  • Body explains the why — what problem existed, what the fix does, any tradeoffs
  • Co-Authored-By trailer — added automatically when Claude helped write or review the code

Example: Asking Claude for a commit message

Claude — Technology Services IT Training
You
Write a commit message for my changes — I fixed the password reset flow to correctly handle expired tokens. The old code was comparing the wrong timestamp field so expired tokens were silently passing validation.
Claude

Here’s a commit message for your changes:

fix: correct expired-token check in password reset flow

The reset endpoint was comparing `created_at` against the expiry
window instead of `expires_at`. This caused expired tokens to pass
validation silently while valid tokens occasionally failed.

Fix the field reference and add a dedicated 422 response for
expired tokens so callers receive a clear, actionable error
rather than a generic 400.

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>

Subject line is 51 characters. Body explains what was broken and what the fix does.

From the Terminal CLI — pipe your diff directly to Claude

Instead of describing your changes by hand, pipe the actual diff so Claude can read what changed:

bash Terminal 
# Review staged changes and generate a commit message
git diff --staged | claude "write a git commit message for these changes"

# If you want a one-liner subject only
git diff --staged | claude "write a single-line git commit subject, imperative mood, under 72 chars"

# Include context about the project
git diff --staged | claude "write a commit message for these changes. this is a PowerShell \
  automation script repo for Acadia IT administration"

From VS Code

In VS Code with the Claude extension active, open the Source Control panel (Ctrl+Shift+G) to review your staged diff visually, then switch to the Claude sidebar and ask: "Write a commit message for the changes in the Source Control panel." Claude reads the diff from the workspace and drafts the message. Paste it directly into the commit message field.

Tip
If you run git commit without a -m flag, git opens your editor. You can paste Claude’s message there. Alternatively, use git commit -m "$(git diff --staged | claude 'write a git commit subject')" to generate and commit in one step — but always review the result first.

5.3 Code Review with /code-review

The /code-review skill reads your current git diff — everything staged and unstaged — and returns structured findings organized by severity. It is the fastest way to catch bugs in scripts and configurations before they reach a shared repository.

How to invoke it

CommandWhat it does
/code-reviewDefault review at medium effort — high-confidence bugs and clear improvements
/code-review lowFast scan, only the most obvious issues
/code-review highBroader coverage, may surface less-certain findings
/code-review maxDeep multi-pass review, most thorough
/code-review --commentPost findings as inline comments on a GitHub pull request
/code-review --fixApply the recommended fixes to the working tree automatically

Example /code-review output

Claude — /code-review skill
You
/code-review
Claude

Code Review — 3 findings  1 bug  2 improvements

Bug — src/auth/reset.py, line 47
Token expiry comparison uses created_at instead of expires_at. This allows expired tokens to authenticate successfully. An attacker with a previously-captured token can reset any account indefinitely.

Fix:

- if token.created_at + TOKEN_LIFETIME > now:
+ if token.expires_at > now:

Improvement — src/auth/reset.py, line 63
Bare except Exception swallows all errors including KeyboardInterrupt and database connection failures. Narrow to TokenExpiredError, TokenInvalidError to preserve specific error paths and improve observability.

Improvement — tests/test_reset.py
No test covers the expired-token code path. Consider adding a fixture that sets expires_at = datetime.now() - timedelta(hours=1) and asserts a 422 response is returned.

When to use --fix
/code-review --fix applies Claude’s suggested fixes directly to your working tree. This is convenient for clear mechanical fixes (wrong variable name, missing error handling) but you should still review the diff before committing. Use it on a clean branch so you can compare before and after.

5.4 Using Git History and Blame

When you’ve just inherited a repository, need to audit a recent change, or want to understand why something works a certain way, git history is the primary source of truth. Claude can run the appropriate commands and interpret the results in plain language.

Questions you can ask Claude about your repository

  • “What commits changed the authentication module in the last month?”
  • “Who last modified the config parser and what was their reason?”
  • “Summarize the changes made between v2.0.0 and v2.1.0”
  • “Find any commits that mention ‘hotfix’ or ‘urgent’ in the last 90 days”
  • “What changed in the deploy scripts since the last release?”

Piping git output to Claude

bash Terminal 
# Get a plain-language summary of recent team work
git log --oneline -20 | claude "summarize what the team has been working on based on these commits"

# Understand who owns a file and why it looks the way it does
git blame src/auth/reset.py | claude "who owns the authentication module and what's the history of changes?"

# Decode a specific commit
git show a3f9c12 | claude "explain what this commit does and why it was necessary"

# Compare releases
git log v2.0.0..v2.1.0 --oneline | claude "write release notes for version 2.1.0 based on these commits"

# Find risky changes
git log --all --oneline --since="30 days ago" | claude "flag any commits that might indicate hotfixes, security patches, or production issues"

Understanding blame output

Git blame output is notoriously hard to read at a glance. Claude can take the raw output and give you a clear picture of file ownership, identify which areas have had the most churn, and highlight sections that have been touched by many different contributors (often a sign of technical debt or complex logic).

bash Terminal 
# Analyze ownership patterns across an entire file
git blame --line-porcelain src/network/firewall.py | claude "summarize who owns each section of this file and identify areas of high churn"

# Find the commit that introduced a specific line
git log -S "password_reset_token" --source --all | claude "explain the history of the password reset token feature"
Claude reads full context
When you pipe git output to Claude, it sees the entire history in one pass. You can ask follow-up questions in the same conversation — Claude remembers the log output and can answer questions like "which of those commits were on the main branch?" without re-running the command.

5.5 Resolving Merge Conflicts

Merge conflicts happen when two branches modify the same region of a file differently. Git marks the conflicted sections and pauses the merge for manual resolution. Claude can read the conflict markers, understand both sides, and propose a resolution — often explaining the tradeoffs between options.

Step-by-step workflow

  1. Trigger the merge

    Run git merge feature/timeout-increase (or pull, rebase, cherry-pick). Git stops with a conflict message listing affected files.

  2. Show Claude the conflicted file

    Open the file — Claude sees both the <<<<<<< HEAD section (your changes) and the >>>>>>> section (incoming changes). Paste the relevant section or ask Claude to read the file directly.

  3. Describe what each side should do

    Give Claude context: “Help me resolve this merge conflict — preserve both the upstream rate-limiting logic and my new timeout value for slow networks.”

  4. Review Claude’s proposed resolution

    Claude explains what each side was trying to do and proposes a merged version. Read it carefully — Claude may not know which behavior is correct for your system.

  5. Accept, edit, and complete the merge

    Apply the resolution to the file, remove all conflict markers, then run git add <file> and git merge --continue (or git commit if not in a rebase).

Example conflict and resolution

Claude — merge conflict resolution
You

Help me resolve this merge conflict. I need to keep both the rate limiting from upstream AND my 60-second timeout for slow-network clients:

<<<<<<< HEAD
    timeout = 30
    rate_limit = 100   # upstream: added rate limiting
=======
    timeout = 60       # my change: slow-network support
>>>>>>> feature/slow-network-support
Claude

The conflict is straightforward — your branch only changed timeout while upstream added rate_limit. Both changes are in the same block so git flagged it, but there’s no logical conflict. The resolution is to keep both values:

    timeout = 60       # slow-network support
    rate_limit = 100   # upstream rate limiting

Remove all three conflict marker lines (<<<<<<<, =======, >>>>>>>) and replace the block with the two lines above. Then:

git add src/network/client.py
git merge --continue

Double-check that timeout = 60 is the correct value for your slow-network use case — you’re the expert on the network characteristics this needs to handle.

Always review conflict resolutions
Claude proposes a resolution based on what it can read in the conflict markers. It does not know your system's runtime behavior, performance constraints, or business rules. Always verify that the resolved code does what you intend before committing, especially for logic in authentication, networking, or data processing.

5.6 Accessing Git Through Each Interface

Claude integrates with git differently depending on which interface you’re using. Each has a natural workflow:

Interface How to use git with Claude Notes
Desktop App Ask Claude to run git commands in your project. Claude uses the Bash tool to execute git log, git diff, git blame, and so on, then shows you the output alongside its interpretation. Requires Bash(git *) permission granted. Claude will ask before running any command that modifies the repository.
Terminal CLI Pipe git output directly: git log --oneline -20 | claude "summarize" — or drop into the Claude REPL and ask questions. Claude can also run git commands itself if you give it permission. Most flexible interface for git work. You control exactly what gets piped in and can chain commands freely.
VS Code Use the Source Control panel to stage files visually, then ask Claude in the sidebar for a commit message or to review the diff. The /code-review skill works particularly well here, with file-level context available. Best for file-level review when you want to see diff and Claude’s response side by side. /code-review --comment posts inline PR comments directly to GitHub.
Permissions note
If Claude cannot run a git command because Bash permission hasn’t been granted, it will tell you the exact command to run yourself. You can paste the output back into the conversation and Claude will interpret it. No access is ever required — you can always be the one running the command.

5.7 Claude's Git Safety Rules (Built-In)

Claude enforces a set of git safety rules by default. These protect your repository from destructive operations that are difficult or impossible to undo. Technology Services staff should know these rules so they understand when Claude will pause and ask for explicit confirmation.

Operations Claude will not perform without explicit instruction

⛔
Force push (git push --force)
Rewrites remote history and discards other contributors’ commits. Claude will warn and refuse unless you explicitly confirm.
⛔
Hard reset (git reset --hard)
Permanently discards uncommitted changes. Claude will offer safer alternatives (stash, soft reset) and only proceed if you confirm you want to lose the changes.
⛔
Force delete a branch (git branch -D)
Deletes even unmerged branches without confirmation. Claude prefers -d (safe delete) and will ask before using the force flag.
⛔
Skip pre-commit hooks (--no-verify)
Bypasses linting, tests, and secret-scanning checks. Claude investigates why a hook failed and fixes the underlying problem rather than skipping the check.
⛔
Amend published commits (git commit --amend after push)
Rewrites shared history. Claude prefers creating a new commit to correct a mistake, unless you explicitly ask to amend an unpublished commit.
⚠️
Force push to main or master
Even if you explicitly ask, Claude will issue an additional warning before force-pushing to a main branch, since this affects all collaborators.
Why this matters for Technology Services
Shared repositories for scripts, configurations, and automation tools are especially sensitive — a force-push or hard reset can delete work from the entire team. Claude’s built-in caution means you don’t need to worry about accidentally destroying history in a shared repo.

5.8 Practical Git Prompts for IT Work

A quick-reference library of prompts that Technology Services staff will use repeatedly. Copy, adapt, and build on these in your own workflows.

"Create a .gitignore for a PowerShell project on Windows — include VS Code, JetBrains, and common Windows temp files"

Generates a comprehensive .gitignore covering editor artifacts, OS files, and PowerShell-specific outputs like transcripts and cached credentials.

"Write a PR description for these changes: [paste diff or summary]"

Produces a pull request body with a summary, motivation for the change, and a testing checklist. Paste into GitHub or Bitbucket.

"What is the git command to stash my changes, pull the latest from main, and reapply my stash?"

Useful when you’re mid-task and need to sync with the remote. Claude gives you the exact command sequence and explains each step.

"Explain what this commit does: [paste git show output]"

Translates a raw patch into a plain-language explanation of what the change does and why it was likely needed. Good for code archaeology.

"Set up a git pre-commit hook that runs PSScriptAnalyzer on any staged .ps1 files and blocks the commit if there are errors"

Claude writes the .git/hooks/pre-commit shell script, handles the staged-file filtering, and explains how to make it executable.

"I need to undo the last commit but keep my changes staged. What’s the safest way to do this?"

Claude suggests git reset --soft HEAD~1, explains what it does, confirms your changes will be preserved, and warns against alternatives that would lose work.

"Help me write a CHANGELOG entry for version 2.3.1 based on these commits: [paste git log]"

Categorizes commits into Added, Changed, Fixed, and Removed sections in Keep a Changelog format. Ready to paste into CHANGELOG.md.

"Find any commits in the last 60 days that might have introduced the login error we’re seeing: [paste error]"

Claude suggests the right git log -S or git log --grep command, runs it, and interprets which commits are most likely candidates.

Build a prompt library
Save prompts that work well for your repositories in a shared document. Over time, your team will accumulate a library of git workflows that consistently produce useful output from Claude, reducing ramp-up time for new staff.

Module 5 Summary

Claude is a capable git collaborator: it writes commit messages and PR descriptions, reviews code for bugs with /code-review, interprets history and blame output, guides conflict resolution, and enforces safety rules that protect shared repositories. The key workflow is always: Claude assists, you review, you approve.

The most important rule is the same one that governs all Claude interactions — review before you commit. A commit message that misrepresents a change, or a conflict resolution that silently drops a logic branch, is harder to detect than a code bug. Stay in the loop.
