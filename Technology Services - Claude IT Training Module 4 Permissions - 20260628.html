4.1 How Claude's Permission Model Works

Claude Code is a capable agent that can execute real shell commands, edit files, make network requests, and interact with external services. This power comes with responsibility: before Claude takes any action that could affect your system, it checks whether that action is permitted.

The permission system exists to protect you from three categories of risk:

⚠️

Accidental Changes

Preventing unintended modifications to files, databases, or configurations you did not mean to change.

🔐

Scope Creep

Keeping Claude's actions within the boundaries appropriate for the current task and project.

🛡️

Security Boundaries

Ensuring sensitive operations like destructive commands or network calls require explicit approval.

Three Decision States

Every action Claude might take falls into one of three categories, configured in settings.json:

State Behaviour Configured by
Allow Claude performs this action without asking. You have pre-approved it. The permissions.allow list in settings.json
Ask Claude pauses and displays a permission prompt in the chat before proceeding. You approve or deny in the moment. Default for any action not listed in allow or deny
Deny Claude refuses to perform this action regardless of what you ask. It is permanently blocked. The permissions.deny list in settings.json
Mental Model
Think of the permission model as a three-tier filter. Claude checks the deny list first — if the action matches a deny rule, it stops immediately. Then it checks the allow list — if there is a match, it proceeds without prompting. If neither list matches, it asks you in the moment. Deny always wins over allow.

4.2 The Settings Files

Claude Code reads its configuration from JSON files stored in predictable locations. Understanding the settings hierarchy lets you set global defaults once and override them per-project or per-person as needed.

The Settings Hierarchy

🌎 Global Settings

Path:
C:\Users\USERNAME\
.claude\settings.json

Applies to every project you open on this machine. Use for permissions and preferences that should always be available regardless of project context.

📁 Project Settings

Path:
[Project Root]\
.claude\settings.json

Applies only when working in this project. Overrides global settings. Committed to git — shared with the whole team.

👤 Local Overrides

Path:
[Project Root]\
.claude\settings.local.json

Personal overrides for this project. Same format as settings.json, but not committed to git. Used for your own preferences that should not affect teammates.

Settings are merged in order from lowest to highest priority: global → project → local. If the same permission appears in both global and project settings, the project setting wins.

The settings.json Structure

The file is standard JSON. The two primary keys are permissions and hooks. Permission patterns use a Tool(glob) syntax — the tool name followed by a glob pattern that matches the tool's input.

.claude/settings.json JSON 
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)",
      "Bash(python *)",
      "Read(*)",
      "Write(*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(format *)",
      "Bash(diskpart *)"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --silent"
          }
        ]
      }
    ]
  }
}

Understanding Permission Patterns

Permission patterns follow the format ToolName(pattern). The * wildcard matches any characters. Here are the common tools you will configure permissions for:

Tool Name What It Controls Example Pattern
Bash Shell command execution (PowerShell, cmd, bash) Bash(git *)
Read Reading file contents from disk Read(*)
Write Writing or creating files on disk Write(src/*)
Edit Making targeted edits to existing files Edit(*)
Glob File pattern matching and directory listing Glob(*)
WebFetch Fetching content from URLs WebFetch(https://acadia.ca/*)
mcp__* MCP server tool calls (email, calendar, Drive, etc.) mcp__gmail__send_email(*)

4.3 Common Permissions for IT Work

The following table covers the permissions Technology Services staff most commonly configure. Risk levels reflect the potential impact if Claude uses the permission unexpectedly or incorrectly.

Permission Pattern Use Case Risk
Read any file Read(*) Claude reads your code, configs, and documentation. Nearly always safe to allow globally. Low
Write any file Write(*) Claude creates or overwrites files anywhere on the filesystem. Consider restricting to Write(src/*) for sensitive projects. Medium
Edit any file Edit(*) Claude makes targeted string replacements in existing files. Safer than Write since it only modifies specified sections. Low
All git commands Bash(git *) Commits, branching, history inspection, push/pull. Claude cannot bypass git hooks, so your existing safeguards still apply. Low–Medium
PowerShell Bash(powershell *) Windows administration, automation scripts, registry access. Review carefully — PowerShell can reach nearly anything on Windows. Medium
npm commands Bash(npm *) Installing packages, running build scripts, running tests. Low risk if you trust your package.json scripts. Low
Python scripts Bash(python *) Running data processing, automation, and utility scripts. Risk depends entirely on what the scripts do. Medium
Directory listing Bash(ls *), Bash(dir *) Browsing files and directory structures. Read-only and essentially harmless. Low
Network diagnostics Bash(ping *), Bash(nslookup *), Bash(tracert *) Diagnosing connectivity, resolving DNS, tracing routes. Useful for network troubleshooting workflows. Low
curl / Invoke-WebRequest Bash(curl *), Bash(Invoke-WebRequest *) Making HTTP requests to APIs or downloading resources. Can exfiltrate data if used carelessly — allow only for specific hostnames if possible. Medium
Docker commands Bash(docker *) Container management for development environments. Docker has elevated privileges on the host — use with awareness. Medium
File deletion Bash(Remove-Item *) Deleting files. Restrict to Bash(Remove-Item *.tmp) or similar rather than allowing broad deletion. High
Recursive deletion Never allow Commands like rm -rf or Remove-Item -Recurse -Force applied broadly. Place in the deny list rather than leaving on ask. High
Pattern Scoping Tip
Patterns support glob wildcards. Instead of allowing all Python scripts with Bash(python *), you can narrow the scope to scripts in your project: Bash(python scripts/*). This limits Claude to running scripts only from a known safe location.

Recommended Starter Configuration for Technology Services

The following settings.json provides a balanced starting point for most Technology Services workloads — read access everywhere, write access with prompts for non-source files, standard dev-tool commands pre-approved, and destructive commands blocked:

Recommended starter: C:\Users\USERNAME\.claude\settings.json JSON 
{
  "permissions": {
    "allow": [
      "Read(*)",
      "Edit(*)",
      "Glob(*)",
      "Bash(git *)",
      "Bash(npm *)",
      "Bash(node *)",
      "Bash(python *)",
      "Bash(ls *)",
      "Bash(dir *)",
      "Bash(Get-ChildItem *)",
      "Bash(ping *)",
      "Bash(nslookup *)",
      "Bash(tracert *)",
      "Bash(ipconfig *)",
      "Bash(netstat *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(Remove-Item -Recurse -Force C:\\Windows*)",
      "Bash(Remove-Item -Recurse -Force C:\\Users*)",
      "Bash(format *)",
      "Bash(diskpart *)",
      "Bash(reg delete *)"
    ]
  }
}

4.4 The /config Command

The /config command opens an interactive settings panel inside the Claude chat interface. It is the quickest way to adjust permissions on the fly without manually editing JSON files.

What /config Can Do

  • View all current permission allow and deny lists
  • Toggle individual permissions on or off
  • Choose whether a change applies globally or to the current project only
  • View and modify hook configurations
  • Set Claude's model preference and verbosity level

Live Example — Using /config

Claude Code — helpdesk-portal
You
/config
Claude
Claude Code Settings

Current Permissions — Project Scope (.claude/settings.json)

Allowed:
  ✓  Read(*)
  ✓  Edit(*)
  ✓  Bash(git *)
  ✓  Bash(npm *)

Denied:
  ✗  Bash(rm -rf *)

What would you like to change? You can say things like:
• "Allow PowerShell commands"
• "Deny all curl commands"
• "Move the git permission to global settings"
• "Show me the hook configuration"
You
Allow PowerShell commands for this project only
Claude
I'll add Bash(powershell *) to the allow list in .claude/settings.json (project scope only).

Change preview:
"allow": [ ..., "Bash(powershell *)" ]

Shall I apply this change?
Scope Awareness
Always confirm which settings file Claude is targeting before accepting a configuration change. "Project scope" writes to .claude/settings.json (committed to git, shared with team). "Global scope" writes to C:\Users\USERNAME\.claude\settings.json (personal, applies everywhere). "Local scope" writes to .claude/settings.local.json (personal, not committed).

4.5 The update-config Skill

The /update-config skill is a specialized skill that handles settings and configuration tasks via natural language. While /config provides an interactive panel, /update-config is better suited for complex configuration tasks — especially setting up hooks, which require precise JSON structure.

When to Use /update-config vs /config

Use /config when… Use /update-config when…
You want to quickly toggle a permission on or off You want to describe a behaviour in plain English and let Claude figure out the correct JSON
You are browsing current settings You are setting up hooks (complex JSON structure)
You need a simple allow/deny change You want to move permissions between global/project/local scope
You know the exact permission pattern You know the behaviour you want but not the exact pattern

Example /update-config Requests

Claude Code — helpdesk-portal
You
/update-config Set up a hook that runs our linting script whenever Claude writes a file, and another that shows a desktop notification when Claude finishes a long task.
Claude
I'll set up two hooks in your project's .claude/settings.json:

1. PostToolUse hook on Write — runs npm run lint --silent after every file write
2. Stop hook — sends a Windows toast notification using msg * "Claude has finished"

Here is the configuration I will add to .claude/settings.json:

"hooks": { "PostToolUse": [{ "matcher": "Write", "hooks": [{ "type": "command", "command": "npm run lint --silent" }] }], "Stop": [{ "hooks": [{ "type": "command", "command": "msg * \"Claude has finished\"" }] }] }
Shall I apply this change?

Natural Language Request Examples

  • /update-config Allow all npm commands — adds Bash(npm *) to the allow list
  • /update-config Block all file writes outside the src directory — adds a deny rule scoped to Write outside src/*
  • /update-config Move the git permission to global settings — removes from project and adds to ~/.claude/settings.json
  • /update-config Run tests before Claude makes any commit — sets up a PreToolUse hook on Bash(git commit *)
  • /update-config Show me all my current hooks — reads and displays the hooks configuration in a readable format

4.6 Hooks — Automated Behaviors

Hooks are shell commands that run automatically at specific points in Claude's workflow — before a tool runs, after a tool completes, or when Claude's response ends. They are the mechanism for implementing "from now on, whenever Claude does X, also do Y" behaviours.

Hook Event Types

Event When It Fires Common Uses
PreToolUse Before Claude runs any tool call Run tests before committing, log tool usage, validate that a command is safe before executing it
PostToolUse After a tool call completes successfully Run linting after file writes, sync files after edits, send audit log entries after bash commands
Stop When Claude finishes generating its response Desktop notifications, logging, summarising what Claude did, triggering downstream processes
Notification When Claude generates an internal notification Routing Claude's internal status updates to external systems

Hook Structure

Each hook has a matcher (which tool or event to watch) and one or more hook definitions (commands to run). The matcher accepts the same tool name patterns used in permissions.

.claude/settings.json — Full hooks example JSON 
{
  "permissions": {
    "allow": ["Read(*)", "Edit(*)", "Write(*)", "Bash(git *)"]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo Running: $CLAUDE_TOOL_INPUT"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --silent"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "powershell -Command \"[System.Windows.Forms.MessageBox]::Show('Claude has finished')\""
          }
        ]
      }
    ]
  }
}

Available Hook Environment Variables

Claude passes context to hook commands through environment variables you can use in your hook scripts:

Variable Contains Example Value
$CLAUDE_TOOL_NAME The name of the tool that fired the hook Write
$CLAUDE_TOOL_INPUT The full input passed to the tool (JSON string) {"file_path": "src/app.js"}
$CLAUDE_TOOL_OUTPUT The tool's result (PostToolUse only) File written successfully
$CLAUDE_PROJECT_DIR The current project root directory C:\Projects\helpdesk-portal

Practical Hook Recipes for Technology Services

📃 Auto-lint on File Write

Runs your project's linter automatically every time Claude edits source code. Catches style violations before they reach git.

"PostToolUse": [{
  "matcher": "Write",
  "hooks": [{
    "type": "command",
    "command": "npm run lint --silent"
  }]
}]

✅ Run Tests Before Commit

Ensures the test suite passes before Claude is allowed to commit. If tests fail, the commit is blocked and Claude is informed.

"PreToolUse": [{
  "matcher": "Bash(git commit*)",
  "hooks": [{
    "type": "command",
    "command": "npm test -- --run"
  }]
}]

🔔 Desktop Notification

Pops a Windows notification when Claude finishes a long task, so you can step away and come back when done.

"Stop": [{
  "hooks": [{
    "type": "command",
    "command": "powershell -Command Add-Type -AssemblyName System.Windows.Forms; [System.Windows.Forms.MessageBox]::Show('Claude finished')"
  }]
}]

📝 Audit Log

Appends a timestamped entry to a log file every time Claude executes a shell command. Useful for IT compliance and change tracking.

"PostToolUse": [{
  "matcher": "Bash",
  "hooks": [{
    "type": "command",
    "command": "echo %date% %time% $CLAUDE_TOOL_INPUT >> C:\\Logs\\claude-audit.log"
  }]
}]
Hook Failures
If a hook command exits with a non-zero exit code, Claude treats this as a signal that the preceding action should be blocked or reviewed. Design hook commands carefully: a failing linter should exit non-zero to block the commit, but a failing notification script should exit zero so it does not interrupt Claude's workflow unexpectedly.

4.7 Permission Best Practices

The Principle of Least Privilege

Grant Claude the minimum permissions it needs to accomplish the task at hand. It is much easier to add a permission when you need it than to audit the damage from an overly permissive configuration.

Never Allow These Globally
The following patterns are dangerous enough that they should never appear in your global allow list, and should explicitly appear in your deny list as a safety net:

Bash(rm -rf *)  —  recursive deletion on Unix/WSL
Bash(Remove-Item -Recurse -Force *)  —  recursive deletion on Windows
Bash(format *)  —  disk formatting
Bash(diskpart *)  —  partition management
Bash(reg delete *)  —  registry deletion

Even if you trust Claude, placing these in the deny list protects you against prompt injection attacks — malicious content in a file Claude reads that attempts to trick Claude into executing destructive commands.
Use Project-Level Settings for Project-Specific Permissions
If a permission is only needed in one project (e.g., Bash(docker *) for your containerised app), put it in that project's .claude/settings.json, not your global settings. This limits the blast radius of any misconfiguration and makes your global settings clean and minimal.
Start Deny-by-Default, Add as Needed
The safest approach is to start with a minimal global settings.json (just Read(*) and Edit(*) allowed) and add permissions as Claude prompts you for them during real work. After a week of normal usage, you will have a permission set that reflects exactly what you actually need — with no unnecessary attack surface.

Personal vs. Shared Permissions

  1. Shared team permissions go in .claude/settings.json (committed to git). These are permissions every team member needs to work on the project — git commands, npm scripts, the project's specific test runner.

  2. Your personal permissions go in .claude/settings.local.json (not committed). Add this file to .gitignore to ensure your personal overrides never accidentally affect teammates.

  3. Machine-wide defaults go in your global ~/.claude/settings.json. Keep this file minimal — it applies to every project, including future ones you haven't created yet.

Adding settings.local.json to .gitignore

.gitignore gitignore 
# Claude Code personal settings — do not commit
.claude/settings.local.json

Regular Permission Audits

Permissions accumulate over time. Schedule a brief quarterly review of your settings.json files to remove permissions that are no longer needed and verify that deny lists still cover the critical cases.

Audit Question Action if "No"
Do I still use every command in my allow list regularly? Remove unused entries
Is my deny list still protecting against destructive commands? Add any missing dangerous patterns
Are project-specific permissions in the project file (not global)? Move permissions to the correct scope
Is settings.local.json in .gitignore for all projects? Add the gitignore entry
Do my hooks still do what I intended when I wrote them? Test each hook manually and update as needed
