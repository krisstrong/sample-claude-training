1.1 What is Claude?

Claude is a large language model (LLM) built by Anthropic, a US AI safety company founded in 2021. Anthropic's research focus is on building AI systems that are reliably safe and understandable — Claude is their flagship product of that mission.

At its core, Claude is a sophisticated text-processing system trained on enormous amounts of human-written text. It understands natural language instructions, generates coherent prose and structured data, writes and debugs code, and reasons through multi-step problems. You interact with it in plain English (or French, or most other languages), and it responds in kind.

Design Principles
Anthropic designed Claude around three properties: Helpful — genuinely useful for real tasks; Harmless — avoiding content that causes harm; Honest — acknowledging uncertainty rather than fabricating confident-sounding answers. These properties shape every response Claude gives.

What Claude understands and generates

  • Natural language text — explanations, summaries, drafts, emails, documentation
  • Code — PowerShell, Python, Bash, SQL, JavaScript, and dozens of other languages
  • Structured data — CSV, JSON, XML, tables, configuration files
  • Analytical reasoning — comparing options, diagnosing problems, evaluating trade-offs
  • Formatted output — Markdown, HTML, numbered lists, step-by-step procedures

Available Claude Models

Anthropic offers Claude in several tiers, each balancing speed, capability, and cost. For Technology Services use, the three you will encounter most often are:

Claude Haiku 4.5
Fast & Lightweight

Best for quick, high-volume tasks: classifying a ticket, extracting a field from a log line, answering a simple factual question. Lowest latency, lowest cost per token.

Claude Sonnet 4.6 Default
Balanced Performance

The everyday workhorse. Strong at scripting, documentation, data analysis, and code review. Most Technology Services tasks start here. This is the model used to build these training materials.

Claude Opus 4.8
Most Capable

Reserved for the most complex reasoning tasks: designing a multi-system architecture, deeply analysing a large codebase, or resolving an ambiguous multi-step problem. Slower and more expensive per token.

Important Limitation
Claude is not connected to the internet by default. It does not browse the web, check live system status, or query external services unless you explicitly provide it with tools that do so. Its knowledge comes from its training data (with a knowledge cutoff date) plus whatever text you paste into the conversation. Always provide relevant context — logs, config files, error messages — rather than assuming Claude already knows your environment.

1.2 How Claude Fits Into IT Work

The most useful mental model for IT professionals is the productivity multiplier: Claude handles the mechanical, time-consuming, or cognitively expensive parts of a task so that you can focus your expertise on judgement, decision-making, and review.

Think of it this way: you still own every output. Claude drafts, you approve. Claude generates the first version of a script, you test and sign off. Claude summarises a pile of tickets, you decide what to escalate. The expertise never leaves the team — Claude just removes the friction around applying it.

💻

Write Code

Generate PowerShell, Python, Bash, and SQL scripts from plain-English descriptions. Debug errors, refactor messy code, add comments and documentation.

📊

Analyse Data

Paste log excerpts, CSV exports, or event viewer output and ask Claude to identify patterns, flag anomalies, or summarise what it sees.

📄

Create Docs

Draft runbooks, KB articles, change requests, incident summaries, and user guides. Claude adapts tone for technical or non-technical audiences.

🔎

Troubleshoot

Describe a problem or paste an error. Claude walks through likely causes, asks clarifying questions, and suggests diagnostic steps.

📊

Review Config

Paste firewall rules, Group Policy settings, or YAML files and ask Claude to check for common misconfigurations or security issues.

🎉

Draft Tickets

Paste raw notes from a support call and ask Claude to structure them into a properly formatted incident or change ticket.

Key Principle
Claude does not replace IT expertise — it amplifies it. A junior team member with Claude still needs an experienced reviewer. An experienced team member with Claude can move significantly faster and produce more consistent outputs. Always review and test Claude's output before applying it in production.

1.4 Key Concepts

You will encounter these terms constantly in Claude documentation, in error messages, and in conversations with colleagues. Understand them now and the rest of the training will make considerably more sense.

Term Definition Practical Implication
Context Window The total amount of text Claude can "see" at one time — both your inputs and its own outputs count against this limit. Think of it as Claude's short-term working memory. Very long conversations or very large file attachments can approach the limit. Break big tasks into chunks, or start a fresh conversation for a new topic.
Conversation A session in which Claude maintains context of everything that has been said so far. Each new conversation starts with a blank slate — Claude does not remember previous separate conversations. Within a conversation you can refer to "the script you wrote earlier." In a new conversation you must re-provide that context.
Prompt Your message or instruction to Claude. Prompts can be a single sentence or several paragraphs with pasted file contents, examples, and explicit instructions. The quality of your prompt directly determines the quality of Claude's response. More context and clearer instructions consistently produce better output. Later modules cover prompt writing in depth.
Token The unit Claude uses to measure text. Roughly three-quarters of a word, or about four characters of English text. A typical paragraph is around 100 tokens. Context limits and plan usage are measured in tokens. A 200 KB log file pasted directly will consume a large portion of a typical context window.
System Prompt A set of instructions provided to Claude before the conversation begins, shaping its persona, tone, and constraints for the entire session. Used by developers and administrators to customise behaviour. In the desktop app, Projects can have a system prompt. In the CLI, you can pass a system prompt via flags. This is how you bake in IT-specific context ("You are a Tier 2 support analyst at a Canadian university") that persists across every message.
Model The specific version of Claude being used — Haiku, Sonnet, or Opus, each with different capability and speed trade-offs. You select the model in the desktop app's model picker, with the --model flag in the CLI, or by setting a default in VS Code. When in doubt, start with Sonnet.
Memory vs Context
A common point of confusion: Claude does not have persistent memory between separate conversations. What it does have is excellent recall of everything within the current conversation's context window. Use Projects (desktop app) or well-crafted system prompts (CLI/API) when you need consistent context across multiple sessions.

1.5 Claude Conversation Example

Here is a representative IT conversation showing how Claude handles a real scripting request. Notice how it immediately produces a working script, explains its choices, and suggests improvements — without needing to be asked for each part separately.

Claude — Technology Services Workspace
You
Write a PowerShell script to list all services that have failed in the last 24 hours and export the results to a CSV file. I want the service name, display name, status, and the last time the status changed.
Claude

Here's a PowerShell script that queries the Windows event log for service control manager events and correlates them with current service status:

PowerShell — Export-FailedServices.ps1 Script 
#Requires -RunAsAdministrator
<#
.SYNOPSIS
    Lists services that have failed in the last 24 hours and exports to CSV.
.OUTPUTS
    CSV file at the path specified by $OutputPath
#>

param(
    [string]$OutputPath = ".\Failed-Services-$(Get-Date -Format 'yyyyMMdd-HHmm').csv",
    [int]$HoursBack = 24
)

$cutoff = (Get-Date).AddHours(-$HoursBack)

# Query Service Control Manager events for failures (Event ID 7034/7031)
$failEvents = Get-WinEvent -FilterHashtable @{
    LogName   = 'System'
    Id        = 7034, 7031
    StartTime = $cutoff
} -ErrorAction SilentlyContinue

$results = foreach ($event in $failEvents) {
    $svcName = ($event.Properties[0].Value -as [string])
    $svc     = Get-Service -Name $svcName -ErrorAction SilentlyContinue

    [PSCustomObject]@{
        ServiceName  = $svcName
        DisplayName  = if ($svc) { $svc.DisplayName } else { 'N/A' }
        CurrentStatus = if ($svc) { $svc.Status }     else { 'Not Found' }
        EventTime    = $event.TimeCreated
        EventId      = $event.Id
        EventMessage = $event.Message -replace '\r?\n',' '
    }
}

if ($results) {
    $results | Export-Csv -Path $OutputPath -NoTypeInformation -Encoding UTF8
    Write-Host "Exported $($results.Count) events to $OutputPath" -ForegroundColor Green
    $results | Format-Table -AutoSize
} else {
    Write-Host "No service failures found in the last $HoursBack hours." -ForegroundColor Yellow
}

How to use it:

  • Run as Administrator (required for WinEvent log access)
  • .\Export-FailedServices.ps1 — default 24 hours, auto-named CSV in current directory
  • .\Export-FailedServices.ps1 -HoursBack 48 -OutputPath "C:\Logs\failures.csv" — custom range and path

Note: Event IDs 7034 (unexpected termination) and 7031 (terminated on crash) cover most failure scenarios. Want me to add 7023 (terminated with error) or filter to specific service names?

You
Yes — add 7023 and also add a column for whether the service is set to auto-restart on failure.
Claude

Done. I've added event ID 7023 to the filter and queried the service's failure recovery configuration via sc.exe qfailure to populate an AutoRestart column. I'll include the updated script in my next message — just say the word.

Example conversation — Claude produces a working, parameterised script, explains its design choices, and offers to refine based on follow-up questions. Notice it does not wait to be asked for clarification — it proactively flags the event ID options.

1.6 Claude Plans for Acadia

Claude is offered through Anthropic's subscription tiers. Understanding the tiers helps you choose the right access level for individual staff members and departmental deployments.

Free Tier

Personal use

Limited daily usage, access to Sonnet and Haiku models, no persistent Projects. Suitable for occasional exploration and personal learning. Not recommended for production IT work — usage limits will interrupt workflows.

Pro Plan

Recommended — Power users

Significantly higher usage limits, priority access during peak periods, access to all models including Opus 4.8, full Projects support. Recommended for individual Technology Services staff members who use Claude daily.

Teams Plan

Recommended — Departments

Shared workspace with team-level Projects, centralised billing, higher context windows, and admin controls for managing members. The right choice for a Technology Services shared workspace where multiple staff collaborate on the same Projects.

Enterprise Plan

Large deployments

SSO integration (SAML/OIDC), advanced admin controls, audit logging, custom data retention policies, and dedicated support. Appropriate if Acadia pursues institution-wide deployment or needs compliance-level data governance.

Getting Access
Contact Technology Services to arrange departmental or individual Pro/Teams access. Do not use personal free accounts for work that involves university data, student records, or systems credentials — consult the data classification policy before pasting any sensitive content into Claude.

1.7 What Claude Can and Cannot Do

Setting accurate expectations is critical. Claude is remarkably capable in many areas, but it has genuine limitations that matter for IT work. Knowing both sides prevents both under-use (not reaching for it when it would help) and over-trust (accepting output without appropriate validation).

Claude CAN do this Claude CANNOT do this (by default)
Write, debug, and explain code in most languages Access the internet, browse websites, or check live data
Draft technical documentation, runbooks, and KB articles Remember anything from a previous separate conversation
Analyse log files, event exports, and CSV data you paste in Execute code or run commands on any system
Explain concepts — networking, security, Group Policy, Active Directory Connect to Acadia systems, databases, or network resources
Review configuration files for common issues and misconfigurations Guarantee its output is correct — all code must be tested
Summarise and categorise support tickets from pasted text Access files on your computer unless you explicitly paste or attach them
Compare options and recommend approaches based on provided context Provide real-time pricing, version numbers, or release notes beyond its training cutoff
Transform data formats — JSON to CSV, YAML to TOML, etc. Authenticate to services or act as a user in external systems
Generate realistic test data for development environments Replace human judgement on policy, security approvals, or change management decisions
The Golden Rule
Claude produces drafts and suggestions, not final authoritative outputs. Every script it writes needs to be read and tested. Every document it drafts needs to be reviewed for accuracy. Every recommendation it makes needs to be evaluated against your knowledge of the actual environment. Treat Claude as an expert colleague who does not know your specific organisation — smart enough to produce good starting points, but needing your local knowledge to get to the finish line.

When Claude Makes Mistakes

Claude can confidently state things that are wrong — a phenomenon called hallucination. For IT work this most often appears as:

  • Incorrect or invented PowerShell/Python syntax that looks plausible but does not run
  • Wrong default values for configuration settings
  • Outdated information about software versions, deprecated APIs, or changed behaviour
  • Plausible-sounding but fabricated event IDs, registry keys, or file paths

The remedy is always the same: test in a non-production environment first, verify against official documentation, and use your expertise to catch what looks off. Claude is aware of this limitation and will often flag uncertainty — but not always, so do not rely on it to self-report errors.
